# 开发实现流程

每个任务开发前，必须确保 `/dev-doc` 流程已完成且评审通过。

## 1. 前置检查

- [ ] 开发文档 (README.md + context.md) 已评审通过
- [ ] 评审守则已整合到 README (如有)
- [ ] context.md 的优先级排序已确认

## 2. 上下游联动检测 (开发前)

开发前必须扫描上下游模块:

### 2a. 去桩对接
- 检查上游模块是否对本任务有 mock/stub/TODO 标记
- 本任务实现后，必须把上游的桩替换为真实对接
- 搜索方式: `grep -r "TODO.*T{任务号}" muse/ phase2/`

### 2b. 文档等待清算
- 检查其他任务文档中是否有 "⏳ 待 T{本任务}" 的标注
- 本任务完成后，更新对应文档的验证边界表 (⏳ → ✅)
- 搜索方式: `grep -r "T{任务号}" phase2/*/README.md`

### 2c. 整体影响评估
- 本任务的实现是否修改了共享模块 (orchestrator/engine/config)？
- 修改会不会影响已完成任务的行为？
- 如有冲突: 先跑全量测试 `node --test muse/` 确认不退化

## 3. 实现顺序

按 context.md 优先级: **P0 → P1 → P2**
每完成一个 P0 子任务就 git commit，不要攒到最后。

## 4. 代码规范

- ESM only (`.mjs` 后缀)
- `createLogger('模块名')` 统一日志
- try/catch 降级，不静默吞错
- 错误场景必须有 log.error
- 不在 catch 里 `return` 空值而不记录

## 5. 模块测试 (立即写)

每完成一个模块，**立即**写对应 unit test:
- 不是全部写完再补测试
- 测试文件: `xxx.test.mjs`，用 `node:test` + `assert/strict`
- 跑通后再开始下一个模块

## 6. 集成测试

模块间协作验证:
- 模块 A 调模块 B → 验证接口兼容
- 配置注册 → 验证加载成功

## 7. E2E 测试

吸取 T10 教训:
- **不依赖外部网络** (models.dev 等不可达时 graceful skip)
- **断言检查日志/文件/工具调用**，不靠模型文本匹配
- **每条 E2E 幂等可重复**
- **明确标注** 哪些是本任务闭环 vs 待后续联动

## 8. 降级验证

异常场景必须覆盖:
- hook/plugin 抛错 → 主链路不受影响
- 网络断 → 不 crash
- 文件写失败 → 降级日志

## 9. 上下游联动检测 (开发后)

### 9a. 去桩清算
- 确认上游的 mock/stub 已替换为真实调用
- 确认下游的 "⏳ 待 T{本任务}" 标注已更新

### 9b. 文档同步
- 实现和文档有偏差 → 更新 context.md
- 新发现的边界/约束 → 更新 README 评审守则
- 上下游 README 的验证边界表 → ⏳ → ✅

### 9c. 全量回归
- 跑 `node --test muse/` 确认不退化
- 确认已完成任务的行为没有被破坏

## 10. 踩坑记录

新增 BUG → 追加到 `phase2/EXPERIENCE.md`:
- BUG 编号连续
- 记录: 现象 / 原因 / 修复 / 教训

## 11. 验收对账

对照 README 验收表逐条打勾:
- 文件层 ✅
- 注册/发现层 ✅
- 行为生效层 ✅
- 降级 ✅

## 12. Git 提交

- 格式: `feat(tXX): 简述` / `fix(tXX): 简述`
- 每完成一个 P0 子任务就提交 (不攒)
- 最终 commit message 包含测试结果摘要
