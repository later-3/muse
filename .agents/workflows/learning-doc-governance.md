# 学习文档治理规则（Learning Doc Governance）

> **适用范围：** `user/foundations/F*.md` + `user/unitXX-*/study/*.md` + `user/unitXX-*/README.md`
> **优先级：** 本文件 > research-note.md > ai-parallel-task.md（本文件是最高约束）
> **核心理念：** Later 的 1 分钟被放大 10 倍。AI 替 Later 读论文、读博客、读源码，深加工后交付高密度内容。

---

## 一、资源治理 — "不允许自己编"

### 1.1 来源金字塔（优先级递减）

```
层级 1: 权威书籍 + 论文          ← 理论根基，必须引用
层级 2: 大学课程 + 大佬博客       ← Stanford/MIT/Berkeley + Karpathy/Weng/Alammar
层级 3: 官方 Cookbook + 文档      ← Anthropic/OpenAI/HuggingFace 的代码
层级 4: 开源项目源码             ← 拆解时必须给具体文件路径
层级 5: AI 自己的解读            ← 只做整合升华，不做原创发明
```

### 1.2 引用规则（强制）

- **每一段核心内容必须标注来源** `[B1]` `[P5]` `[W4]` `[U2]` `[G6]`
- 来源索引见 `user/reference/INDEX.md`
- 具体文件映射见 `user/reference/SOURCE_MAP.md`
- **写一个知识点前，先查 SOURCE_MAP 找素材，不要凭空写**
- 本地有的资源（`repos/`）优先引用本地路径
- **路线/排序/取舍决策必须对照权威课程**（C8/U2/G7/G5/Weng 的教学顺序），不靠感觉

### 1.3 资源获取流程

```
要写某知识点 → 查 SOURCE_MAP.md → 找到本地文件
  ↓ 本地没有？
  → 搜索权威博客/论文 → 能 clone 就 clone 到 repos/
  → 不能 clone → 给链接 + 把关键内容引过来
  → 更新 INDEX.md 和 SOURCE_MAP.md
```

## 二、内容治理 — "第一性原理讲透"

### 2.1 每个知识点的 4 层深度（强制）

```
层 1: 是什么 (What)     ← 一句话定义，带英文原词
层 2: 怎么做到的 (How)   ← 第一性原理 + 数学/代码/流程图
层 3: 为什么有效 (Why)   ← 不能只说"有效"，要说清因果
层 4: 具体例子 (Example) ← 真实例子（不是编的），最好来自论文/项目
```

**反例（不合格）：**
> "在 RL 阶段，reward 函数奖励先推理再回答的行为 → 模型自动学会输出 `<think>` 标签"

**正例（合格）：**
> "**[P5] DeepSeek R1 §2.2:** 对同一个数学题生成 8 个回答，用规则判对错打分。例如鸡兔同笼：回答 A（有 `<think>` 推理）= 正确 = reward 1，回答 D（直接猜）= 错误 = reward 0。组平均 0.5，A 的优势 = 正，D 的优势 = 负。模型学到：先推理 → reward 更高。"

### 2.2 必须有的 3 种验证

| 验证方式 | 要求 |
|---------|------|
| **理论验证** | 引用论文/书籍的具体章节，不能只给结论 |
| **代码验证** | 引用 repos/ 里的具体文件，标注关键行 |
| **项目验证** | 至少 1 个真实项目在用这个原理（Claude Code / Aider / Muse 等） |

### 2.3 禁止的做法

- ❌ AI 自己编原理解释（没有来源的"我的理解"）
- ❌ 只说"有效"不说为什么有效
- ❌ 用类比代替原理（类比只是辅助，不是定义）
- ❌ 代码没有来源标注
- ❌ 论文只给标题不给章节号
- ❌ 低质量填充（一个概念写 3 句就够不要写 3 段）

## 三、格式治理 — "10 倍放大的结构"

### 3.1 文档结构（Foundation F*.md）

```
§0   ⚡ 3 分钟速读版（5行以内，含面试必记金句）
§0b  🔬 核心原理三栏表（概念 × 来源 × 激活方式 × 类比）
§1-N 📖 正文章节（每章先列来源，再讲内容）
     每章必须有:
       > **来源: [XX] 名称 §章节**
       具体内容 + 例子
§N+1 🎯 Muse/OpenCode 映射表
§N+2 💼 面试必答（Q&A 格式，可直接复述）
§N+3 ✅ 自检题（3-5 道 + 折叠答案）
§N+4 📚 延伸阅读（分层: 入门/进阶/深水区）
```

### 3.2 文档结构（Unit study/*.md）

在 research-note.md 的基础上，**额外要求**：

- §0b 必须有「上下文定位」— 说清楚"这个文档在学习路线里的位置"
- §2b 必须有来源引用（每段标 `[ref-XX]`）
- §4 开源项目映射 → **必须引用 repos/ 里的代码**，给出文件路径
- §7 面试题 → **必须有"故事题"**（用 Muse 项目讲一个 STAR 故事）

### 3.3 文档结构（Unit README.md）

```
§1 学习目标（3-5 个问句）
§2 前置基础查表
§3 📖 学习文档表（study/ → 状态）
§4 🤖 AI 并行任务表（experiments/ → 结果）
§5 🔧 OC 实战任务（3 类: 理论实操 / 课程练习 / 项目拆解）
§6 🏗️ 并行项目里程碑（Muse + 学习助手）
```

### 3.4 文档结构（Unit experiments/ 任务清单）

> **核心要求：** 新 AI 打开 README 的 `🤖 AI 并行任务` 就知道"接下来做什么"。

每个 Unit README 的 `§4 🤖 AI 并行任务` 必须包含：

```
### 已完成
| 类型 | 文件 | 结果 |
|------|------|------|
| 🧪 实验 | `exp01-xxx.mjs` | X/X ✅ |
| 🔧 消险 | `R1-xxx.md` | 评估 X/10 |

### 待做（按优先级排序）
| # | 类型 | 实验名 | 验证什么 | 对应理论 | 前置条件 | 状态 |
|---|------|--------|---------|---------|---------|------|
| exp02 | 🧪 | xxx | xxx | 01a §X | exp01 完成 | [ ] |
| R2 | 🔧 | xxx | xxx | — | — | [ ] |
```

**规则：**
1. **待做清单必须存在** — 不能只列已完成的。新 AI 看到 `[ ]` 就知道从哪开始
2. **每个实验必须写"验证什么"** — 一句话说清假设，不是只写文件名
3. **标注前置条件** — 哪些实验依赖其他实验先完成
4. **标注对应理论** — 指向 study 文档的具体章节
5. **实验 vs OC 的区别：** 实验 = AI 先做验证（Later 吸收即可），OC = Later 自己动手做（学了就练）
6. **API 和环境需求** — 如果实验需要 LLM API key 或特殊依赖，在 README 的前置基础里注明

## 四、效率治理 — "放大 1 分钟"

### 4.1 信息密度要求

| 原则 | 具体做法 |
|------|---------|
| **高价值优先** | 面试必考 > 原理深挖 > 工程细节 > 历史背景 |
| **去除低质量** | 删除废话、删除重复、删除"众所周知"类空话 |
| **一句话能说清就不要一段** | 概念定义用 1 句话 + 英文原词 |
| **表格优于段落** | 对比类信息必须用表格 |
| **代码优于文字** | 能用代码/伪代码说明的不用文字描述 |
| **折叠非核心** | 补充信息用 `<details>` 折叠 |

### 4.2 Later 的阅读路径

```
赶时间（3分钟）: 只看 §0 速读版
正常学习（15分钟）: §0 + §1-N 正文 + §面试必答
深入学习（30分钟）: 全文 + 打开 repos/ 看代码
复习模式: 只看面试必答 + 自检题
```

### 4.3 AI 的工作分配

| AI 做（低价值重复劳动） | Later 做（高价值判断） |
|---------------------|-------------------|
| 读论文摘取关键段落 | 判断哪些真正有用 |
| 读源码标注关键行 | 决定 Muse 怎么借鉴 |
| 整理对比表 | 面试时怎么讲故事 |
| 写自检题和答案 | 做自检题验证理解 |
| 搜索优质资源 | 决定学习优先级 |

## 五、关联治理 — "逻辑完备"

### 5.1 每个文档必须标注的 3 个关联

```
上游依赖: "看这个之前要先看 FX / unitXX"
下游去向: "看完这个可以看 FX / unitXX"
OC 实战: "这个知识点对应 ocXX 任务"
```

### 5.2 三线并行的关联

| 线 | 内容 | 关联规则 |
|----|------|---------|
| **主线项目（Muse）** | 每个知识点 → "Muse 哪里用了？怎么改进？" | §Muse 映射表 |
| **支线项目（学习助手）** | 每个知识点 → "学习助手哪个版本用到？" | §项目里程碑 |
| **OC 实战** | 每个知识点 → "哪个 oc 任务直接练这个？" | README OC 表 |

### 5.3 面试关联（强制）

- 每个 F 文档至少 3 道面试题
- 每个 unit study 至少 2 道面试题
- 面试题类型必须覆盖：基础概念 / 设计题 / 故事题（Muse）/ 对比题
- **目标：学完之后 = AI Agent 技术大佬**

### 5.4 整体路线逻辑验证

```
因果链（每一层依赖前一层，没有跳步）:
  F1-F3   → 理解 LLM 是什么（预测机器 + 训练 + 思考）
  unit01  → 理解 Agent 是什么（LLM + 循环 + 工具）
  unit02  → 理解多 Agent 协作（编排 + Handoff + 评估）
  unit03  → 理解持久化（状态机 + 记忆 + 向量搜索）
  unit04  → 理解 Prompt 设计（7层 + 参数 + 格式）
  F9-F10  → 理解定制+部署（LoRA + 量化 + 3090）
  Muse    → 全部理论的实战载体
  学助     → 第二个验证场
```

## 六、质量检查清单

**资源**
- [ ] 每个核心知识点都有 `[ref-XX]` 来源标注？
- [ ] 代码示例来自 repos/ 并标注了文件路径？
- [ ] 没有"AI 自己编"的原理解释？

**内容**
- [ ] 每个概念都有 4 层深度（是什么/怎么做到/为什么/例子）？
- [ ] 用第一性原理讲透了，不是表面定义？
- [ ] 去除了低质量填充？

**格式**
- [ ] 有 3 分钟速读版？
- [ ] 有核心原理三栏表？
- [ ] 有面试必答（至少 3 题）？
- [ ] 有自检题（3-5 题 + 折叠答案）？
- [ ] 表格优于段落、代码优于文字？

**关联**
- [ ] 标注了上游依赖和下游去向？
- [ ] 有 Muse 映射？
- [ ] 有对应的 OC 实战任务？
- [ ] 面试题覆盖了基础+设计+故事+对比？

**效率**
- [ ] Later 花 15 分钟能吸收核心？
- [ ] 总行数 ≤ 800 行？
- [ ] 非核心内容已折叠？

## 七、文档迭代规则

```
版本 [占位]  = 框架+占位内容，标注 [TODO]
版本 [AI✓]   = AI 完成初稿，等 Later 审阅
版本 [Later✓]= Later 确认，可用于面试/学习
版本 [升级]  = 找到了更好的来源，需要重写
```

**触发升级的条件：**
- 找到了更权威的来源（新论文/新课程/更好的博客）
- 发现内容有事实错误
- Later 反馈"这段不够透彻"
- 新 clone 了相关 repo，有更好的代码示例
