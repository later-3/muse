# 真实数据验证流程

> 源自 BUG-109 教训: 归纳 ≠ 理解机制。150 个测试通过但用的是自编数据 = 自证循环。

本流程在 `/dev-impl` 的 **测试阶段之后、提交之前** 执行。
目的: 确保你的代码在**真实环境**中的行为和你的假设一致。

## 1. 外部数据格式验证

如果你的代码依赖**外部系统的输出格式** (OpenCode API、MCP 工具名、第三方响应结构):

- [ ] **读源码/文档**: 找到外部系统生成该格式的源代码或文档，理解**机制**而不是从样本归纳
  - 例: OpenCode 工具命名 → 找 `{serverName}_{toolName}` 拼接逻辑
  - 例: API 响应字段 → 找 zod schema 或 TypeScript 类型定义
- [ ] **多样本验证**: 至少查 3 个不同场景的真实输出
  - 不同 MCP server → 不同工具名格式
  - 不同 session → 不同工具组合
  - 不同消息类型 → 不同 parts 结构
- [ ] **边界场景**: 找到最不典型的输入
  - server 名含连字符 (`memory-server`)
  - 工具名含特殊字符
  - 空工具调用的消息

## 2. 输入类型全覆盖

如果你的代码对输入做**分类/路由/解析**:

- [ ] **列举所有实际输入类型**: 不是你能想到的类型，是**系统中实际存在**的类型
  - 例: T17 分类 → 列举 opencode.json 中配置的所有 MCP server 的真实工具名
  - 例: 感知分类 → 列举 Telegram 实际发过来的所有消息类型
- [ ] **每种类型至少一个真实测试用例**: 用真实数据而不是自编数据
- [ ] **"unknown" 路径必须有兜底**: 如果出现意外输入，行为是什么？是否有日志？

## 3. 主链集成验证

如果你的模块需要**被其他模块调用**才能产生价值:

- [ ] **grep 生产调用点**: `grep -r '你的函数名' muse/ --include='*.mjs'`，过滤掉 test 文件
  - 如果只在 test 和模块自身出现 → **没接入，不算完成**
- [ ] **端到端触发**: 通过真实用户入口 (Telegram/Web) 触发一次，检查你的模块是否产生了预期效果
  - 例: 发一条消息 → 检查 /api/executions 是否有记录
  - 例: 发一张图片 → 检查 /api/gaps 是否有记录

## 4. 审核检查项 (给审核人)

审核时除了看代码质量，还要问:

- [ ] "你的分类规则/解析规则是从**哪里**来的？是看了源码还是猜的？"
- [ ] "你的测试用例的输入是**真实数据**还是自己编的？"
- [ ] "你 grep 了生产代码确认你的模块确实被调用了吗？"

## 使用方式

```
/dev-verify
```

在 `/dev-impl` 流程的第 7 步 (E2E 测试) 之后、第 12 步 (Git 提交) 之前执行。
