# 开发文档编写流程

每个任务 (如 T10, T10.5, T11 等) 的开发文档由两个文件组成：
- `README.md` — 概览 (背景/定位/目标/验收/风险预警)
- `context.md` — 详细技术方案 (供评审和开发参考)

## README.md 必须包含

1. **定位** — 这个任务在 Muse 架构中的角色，它不是什么 (划清边界)
2. **背景** — Phase 1 / 现状的什么问题催生了这个任务
3. **与其他任务的关系** — 上游依赖 + 下游消费者，用表格列出具体边界
4. **目标** — 分阶段的具体交付物
5. **验收标准** — 三层验证: 文件层 → 发现/注册层 → 行为生效层
6. **评审守则** — 评审后落地的关键约束 (如有)

## context.md 必须包含以下 11 个维度

### 一、需求背景
- Phase 1 / 现状缺少什么
- 用代码片段展示现有问题
- 为什么需要这个任务

### 二、目标
- 子任务拆分表 (编号 + 交付物)
- 分阶段: 先做什么后做什么

### 三、与 Muse 整体的对齐
- 和 ARCHITECTURE.md 的映射
- 和 PHILOSOPHY.md 原则的关系
- 和 Muse 长期路线 (Phase 2→5) 的衔接

### 四、与前后任务的关系
- 上游任务: 依赖什么、它们提供了什么
- 下游任务: 谁消费本任务的产出
- 并行任务: 需要协调什么

### 五、OpenCode 源码分析
- 从源码实证关键机制 (标注文件名 + 行号)
- 不要猜 API 行为，看 zod schema / 源码

### 六、技术方案
- 目录结构
- 核心数据结构 / 接口设计
- 关键流程图 (文字或 mermaid)

### 七、详细设计
- 每个子模块的详细设计
- 输入 / 输出 / 安全边界 / 伪代码

### 八、优先级排序
- 哪些先做 (MVP)，哪些预留
- 优先级依据

### 九、风险与降级
- 每个风险: 等级 + 对策
- 标注"当前低但演进中会升高"的风险

### 十、测试方案
- 单元测试: 测什么、怎么测
- 集成测试: 模块间协作验证
- E2E 测试: 端到端验证
- **E2E 要求** (吸取 T10 教训):
  - 不依赖外部网络 (不走 models.dev 等)
  - 断言检查工具调用日志 / 事件记录，不靠模型文本输出匹配
  - 明确区分"本任务验收"和"等后续任务联动验收"
  - 每条 E2E 必须可重复执行 (幂等)

### 十一、验收标准
- 三层验证: 文件/注册 → 发现/可见 → 行为生效
- 明确哪些是本任务闭环，哪些待后续任务坐实
- **结论必须标注证据等级**: 已证实(代码/测试) / 推断(文档/设计) / 未验证。高风险结论（"已实现""E2E 通过""与架构一致"）必须附具体文件或测试引用

### 十二、Agent 设计合理性 (参照 AGENT_DESIGN.md)

- 本任务属于工程层还是认知层？如果是认知层，是否在暴露能力而非替代 AI 决策？
- 按问题类型，选择了正确的扩展层吗？(AGENTS.md / Skill / MCP / Hook / 源码)
- Agent 完成这个功能后，她知道自己有了这个新能力吗？(AGENTS.md / Skill 是否需要更新)
- 失败时的降级策略是什么？(不是 "报错"，是 "降级到什么行为")
- 对照 AGENT_DESIGN.md §10 模块映射表，本任务受哪些原则约束？

### 十三、可调试性
- 每个模块的错误场景必须能通过日志定位
- 设计时必须回答: "出了问题看哪个日志能知道是哪里的问题?"
- 关键路径必须有 `createLogger` 日志覆盖
- 降级场景的 catch 必须 log.error，不允许空 catch
- 日志必须能区分: 哪个模块、哪个 Hook、哪个工具、哪个 session

## 使用方式

准备写任务开发文档时，调用此 workflow:

```
/dev-doc
```

然后按上述维度逐一填写 context.md。先写完交用户确认，再进入开发。
