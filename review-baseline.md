# Muse 通用评审基线

> 适用范围：
> - 所有 phase 的方案文档评审
> - 所有 phase 的开发结果评审
> - 所有 Muse 相关模块、任务、集成链路评审

## 目的

避免每次评审都依赖临时提醒。  
从现在开始，Muse 的评审默认按这份基线执行。

---

## 一、评审总原则

评审不是只看“这个模块能不能跑”，而是同时看：

1. 它是否符合 Muse 整体目标
2. 它是否符合当前 Phase 目标
3. 它是否和前后任务、已有代码、未来演进一致
4. 它是否是真落地，不是表面工程或伪实现
5. 它是否有足够强的测试和证据支撑

---

## 二、默认评审维度

以后默认按以下 `10` 个维度评审。

### 1. 目标对齐

检查点：

- 是否符合 Muse 总目标
- 是否符合当前 Phase 目标
- 是否符合当前任务的定位
- 是否偏离长期路线

关注：

- 有没有做成“局部看起来对，但整体方向偏了”

### 2. 上下游关系

检查点：

- 和前置任务是否衔接正确
- 和后续任务是否留好了接口和边界
- 和现有模块是否兼容
- 是否会导致后面返工

关注：

- 不能孤立看模块，必须拉通前后文

### 3. 架构边界

检查点：

- 这一层该做什么
- 不该做什么
- 有没有越层、缺层、职责漂移

关注：

- 人格、策略、能力、生命周期、适配器这些层不能混

### 4. 实现完整性

检查点：

- 是真实实现还是占位
- 是核心逻辑落地还是只做表面接口
- 有没有伪实现、空实现、假联调

关注：

- 文档写得很好不等于代码真实具备能力

### 5. 代码与设计质量

检查点：

- 结构是否清晰
- 抽象是否合理
- 命名是否准确
- 是否容易维护和演进
- 有没有明显技术债或买坑

关注：

- 不是只看能跑，还要看以后会不会烂掉

### 6. 测试与证据

检查点：

- 单元测试是否覆盖核心逻辑
- 集成测试是否覆盖关键链路
- E2E 是否真能证明行为
- 手工验证是否有可归档证据
- 测试断言是否足够强
- **结论是否标注了证据等级**: 已证实(代码+测试) / 推断(文档/设计) / 未验证

关注：

- "测试是绿的" 不等于 "验证是充分的"
- **不允许把规划写成现状** — "已实现/已支持"必须有代码证据
- **不允许把弱验证写成已验证** — skip/mock/弱断言不等于"E2E 通过"
- **不确定就写"未验证"** — 不允许为了完整而脑补”

### 7. 风险与技术债

检查点：

- 当前有哪些隐患
- 哪些是现在必须修
- 哪些可以后续阶段优化
- 哪些会在后面被放大

关注：

- 区分 blocker、当前可接受风险、长期技术债

### 8. 文档-代码一致性

检查点：

- 文档承诺的内容，代码是否真的实现
- 代码已经实现的内容，文档是否同步更新
- 文档里的接口、字段、测试数、状态、路径、命名是否和代码一致
- 开发后，是否把前置文档、总览文档、联动文档一起更新

重点关注 `4` 类错位：

1. **文档有，代码没有**
   - 方案写了，实际没做
2. **代码有，文档没有**
   - 开发改了，但设计和说明还停在旧版本
3. **局部更新了，联动文档没更新**
   - 比如任务 README 更新了，但 phase 总览、上下游文档没更新
4. **前后口径不一致**
   - 前面文档还是旧说法，后面代码已按新方案实现

这是 Muse 项目里的默认硬性评审项，不再省略。

### 9. 可调试性与可观测性

检查点：

- 出问题时，是否能通过日志、状态、错误信息快速定位到模块和阶段
- 关键链路是否有足够的诊断信号
- 日志是否结构化、可搜索、可关联
- 错误是否有上下文，而不是只返回模糊报错
- 是否区分正常信息、警告、异常、降级状态

重点关注 `4` 类问题：

1. **出了问题不知道卡在哪**
   - 只能看到“失败了”，但不知道是配置、网络、模型、工具、MCP、Hook 还是业务逻辑
2. **日志太少**
   - 没有关键阶段日志，无法定位链路断点
3. **日志太乱**
   - 全是无结构输出，无法按 session、task、request、tool 关联
4. **错误不可操作**
   - 报错看完也不知道下一步查什么

Muse 默认不接受“出了问题很难查”的实现。  
评审时必须检查：**出事后，开发者是否能靠日志和状态快速判断问题层级。**

### 10. 结论与建议

输出时必须明确：

1. 是否通过
2. 是否可继续推进
3. 哪些问题必须现在修
4. 哪些问题可以后续阶段优化
5. 下一步最值得做什么

---

## 三、文档方案评审时的额外要求

审方案时，重点看：

1. 目标是否对
2. 边界是否清楚
3. 和前后任务是否联动
4. 验收标准是否可验证
5. 是否为未来演进留出空间
6. 是否和现有真实代码风格、接口、约束一致

不能只看“写得通不通顺”。

---

## 四、开发结果评审时的额外要求

审实现时，重点看：

1. 是否按方案真实落地
2. 是否出现伪实现或空实现
3. 测试是否证明了关键行为
4. 是否把相关文档同步更新
5. 是否对整体系统造成新的断层
6. 是否具备足够的可调试性，出了问题能否快速定位

尤其要检查：

- 模块级是绿的，但系统级是否真的通
- README/context/review 文档是否还停留在旧口径

---

## 五、默认输出风格

以后默认评审输出应包含：

1. 摘要
2. 主要发现（按严重度排序）
3. 整体判断
4. 下一步建议

如果是代码评审：

- Findings 优先
- 总结其次

如果没有发现问题，也要明确说：

- 当前未发现 blocker
- 剩余风险或验证缺口是什么

---

## 六、对我自己的执行要求

从现在开始，除非用户明确要求缩小范围，否则我默认：

1. 先看整体，再看局部
2. 先看架构和目标，再看代码细节
3. 默认检查文档与代码一致性
4. 默认检查前后任务联动关系
5. 默认检查可调试性与诊断信号是否够用
6. 默认区分“当前必须修”和“后续可优化”

一句话：

> Muse 的评审，不再是“这个点有没有问题”，而是“它在整个系统里是否站得住、说得通、落得实、能演进”。
