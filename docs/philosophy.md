# Muse 开发哲学

> **本文档是 Muse 项目的设计灵魂，所有开发和设计决策必须以此为准绳。**

---

## 核心定位

**Muse 是基于 OpenCode 底座构建的 AI Agent 产品。**

我们不是在写"调 API 的传统软件"，我们是在**开发 Agent**——一个有记忆、有性格、会成长的 AI 灵魂。

---

## 第一原则：AI 是大脑，我们是身体

```
❌ 传统思维: 我的代码做决策，AI 只是我调用的"函数"
✅ Agent 思维: AI 做决策，我给 AI 提供"能力"
```

| | 传统方式 | Agent 方式 |
|---|---|---|
| 决策者 | 我们的 if/else | LLM |
| 意图判断 | 正则匹配 | Sisyphus Intent Gate |
| 模型选择 | 硬编码 switch | opencode.json + Agent 自选 |
| 记忆检索 | 每轮机械查询 | AI 通过 MCP 工具按需回忆 |
| 偏好提取 | 正则 + 关键词 | AI 用 MCP 工具自主存储 |

**判断标准**: 写代码前问自己——"这个决策，AI 能做得比我的 if/else 更好吗？" 如果是，就把能力暴露给 AI，而不是替 AI 做决策。

### 1.1 AI 决策的边界

> **Phase 1 教训**: BUG-009 中 `prompt_async` 不 await，内部错误被静默吞掉。如果把错误处理也"交给 AI"，系统会在无声中失败。

**认知决策 → 交给 AI**:
- 意图判断（闲聊 vs 技术问题）
- 模型选择（简单问题用小模型）
- 记忆检索时机（需要回忆时才查）
- 偏好存储（AI 判断什么值得记住）
- 回复语气和风格

**系统决策 → 工程层负责**:
- 安全边界: API Key 保护、白名单、权限校验
- 资源上限: 超时控制、重试次数、消息长度
- 进程生命周期: 启停、守护、crash recovery
- 数据一致性: SQLite 事务、迁移、备份
- 错误处理: 全局 catch、降级策略、日志记录
- 配置校验: model ID 合法性、schema 验证

**判断口诀**: "AI 管思考，工程管生死。"

### 1.2 感知通道 vs 功能实现

> **T14.5 教训**: 第一版设计中，我们替 Muse 写了一个"图片理解桥"，帮她把图片转成文字再喂给她。这违反了第一原则——我们在替 AI 做认知工作。

**原则**: 工程层只负责建立感知通道，不替 AI 理解内容。

```
❌ 替 AI 做: 我们写 bridge 把图片→文字描述，再发给 AI
✅ 给 AI 通道: 我们下载图片到本地，告诉 AI "你收到了一张图片，在这里"
```

| 我们做（身体/通道） | AI 自己做（大脑/认知） |
|-------|---------|
| 从 Telegram 下载图片保存到本地 | 决定要不要看这张图 |
| 告诉 AI "收到了图片，路径是 X" | 用多模态能力看图、或启动子 Agent、或调工具 |
| 工程上确定的事直接做（比如知道是图片就下载） | 需要理解和判断的事由 AI 做 |
| 不可用的类型告知 AI（如语音暂不支持） | AI 决定如何回应不支持的情况 |

**判断口诀**: "确定的工程活直接做，认知判断让 AI 来。"

### 1.3 Agent 自我成长与能力沉淀

> **设计 Agent 不是实现功能**。我们提供基础感知系统和工具体系，Agent 自己积累经验、总结能力、提升效率。

**成长闭环**:

```
遇到新情况 → AI 探索解决 → 成功 → 沉淀为能力 → 下次直接用
                          → 失败 → 记录经验 → 找其他方案
```

**能力沉淀的形式** (从轻到重):

| 形式 | 适用场景 | 例子 |
|------|---------|------|
| Skill (.md) | 新知识/规则 | "处理图片时用 bash 调 python PIL" |
| Script | 可复用的操作 | 图片压缩脚本、格式转换脚本 |
| MCP Tool | 需要结构化输入输出的能力 | 记忆读写、日程管理 |
| Hook / Plugin | 系统级拦截和修改 | 自动注入身份、消息预处理 |

**谁来总结**: 大脑（AI）自己总结能力。小脑观察模式、触发总结提醒。

**关键**: Muse 必须熟悉自己的架构。她不知道自己有哪些工具、怎么扩展自己，就没法自我优化。AGENTS.md 里应包含她的架构概述和工具清单。

### 1.4 技能是可叠加的，身份是流动的

> **小缪是一个多面的生命体，不是单一角色。**
> 每一种能力都是她的**技能**，不是她的**身份**。
> 就像一个人可以是摄影师也是程序员，小缪可以是伴侣、也可以管理开发项目、也可以分析健康数据。

```
✘ 错误: 让小缪变成"开发者" (限定了执行方式)
✘ 错误: 让小缪变成"项目经理" (限定了身份)
✔ 正确: 给小缪增加「项目开发」技能 (技能是可叠加的)
```

**当小缪需要完成开发任务时**:
- 她不是自己写代码 → 她是**编排者** (orchestrator)
- 她调度 OpenCode agents、oh-my-opencode workers 去执行
- 她审查结果、判断质量、汇报给 Later
- 这和她使用 MCP 工具查记忆、用 Whisper 转语音是一样的 — 都是**调用工具完成任务**

**设计新能力时的检验**:
1. 这个能力是作为**技能**增加的吗？(不是把她变成另一个角色)
2. 她是**编排者**还是**执行者**？(如果是执行者，考虑让她调度工具)
3. 这个技能和她已有的技能能共存吗？(不冲突、不限定)

## 第二原则：站在 OpenCode 的肩膀上

OpenCode 是成熟的 Agent 框架，拥有：

| 能力 | OpenCode 提供 | 我们的做法 |
|------|-------------|-----------|
| 推理 | 多 Provider + 多模型 | **用它**，不自己封装模型切换 |
| 工具 | 44 个内置工具 | **用它**，不自己写文件读取 etc |
| Session | SQLite 持久化 + Compaction | **用它**，不自己管 session 生命周期 |
| 配置 | opencode.json 7 级优先级 | **用它**，模型配置走 opencode.json |
| 扩展 | Plugin / Hook / Skill / MCP | **用它**，通过这四层注入 Muse 能力 |
| Agent | build / plan / explore / sisyphus | **用它**或自定义 Agent |
| 事件 | Bus 47 种事件 + SSE | **用它**，做实时响应 |

**做法**: 不 fork OpenCode 重写。在它的架构上扩展——Plugin 机制、Hook 点位、Tool 注册、MCP 协议、Skill 文档、AGENTS.md。必要时进行**源码级增强**，但始终保持与上游的兼容性。

### 2.1 工程主权边界

> **Phase 1 教训**: BUG-011 中 `/session/status` 的 idle 删除行为只有读源码才能发现。过度依赖 OpenCode 内部实现会让 Muse 极其脆弱。

**原则**: 依附 OpenCode 的能力，不等于放弃 Muse 自身的架构边界。

```
✅ 依赖 OpenCode: Session 管理、工具调用、模型推理、Compaction
✅ 保持适配层: Engine 封装所有 REST API 调用 → OpenCode 升级不影响上层
✅ 关键兜底: MCP 不可用 → 回退到直接查 SQLite 记忆
✅ 数据主权: 记忆数据存 Muse 自己的 SQLite，不依赖 OpenCode session

❌ 不做: 直接在业务层调用 OpenCode 内部 URL
❌ 不做: 假设 OpenCode API 行为不变（必须有 fallback）
❌ 不做: 完全放弃 Orchestrator（保留系统级编排能力）
```

## 第三原则：传统工程做基础设施

Agent 思维不等于抛弃传统技术。**基础设施必须用传统工程做到极致**：

| 层 | 技术 | 态度 |
|----|------|------|
| 数据存储 | SQLite + 迁移 + 索引 + 备份 | **必须扎实**，这是 Muse 的"长期记忆硬盘" |
| 通道适配 | Telegram / Web / CLI | **必须稳定**，这是 Muse 的"嘴巴和耳朵" |
| 进程守护 | 小脑 + launchd | **必须可靠**，Muse 必须永远在线 |
| 配置管理 | .env + 校验 | **必须安全**，API key 等敏感信息 |
| 测试 | node:test 全覆盖 | **必须完整**，每个模块都有用例看护 |

**判断标准**: "这是给 AI 用的能力" → Agent 方式。"这是让系统跑起来的基础" → 传统工程。

---

## 第四原则：四层扩展，不改源码

优先用 OpenCode 原生扩展机制，从轻到重：

```
1. Skill (.md)      — 最轻: 教 AI 新规则/知识，热部署
2. AGENTS.md        — 轻量: 项目级规则注入
3. MCP Server       — 中量: 给 AI 提供外部工具和数据源
4. Plugin + Hook    — 重量: 拦截和修改 OpenCode 生命周期

   ↓ 只有上面都不够时

5. 源码级修改       — 最重: 扩展 OpenCode 核心能力
```

## 第五原则：Muse 的独有价值

OpenCode 不做、但 Muse 必须做的事。区分**产品目标**（用户感知）和**系统能力**（实现手段）：

### 产品目标（WHY — 用户为什么用 Muse）

| 目标 | 用户感受 |
|------|----------|
| **终身陪伴** | "它认识我，知道我要什么" |
| **成长进化** | "越用越懂我，越来越聪明" |
| **情感连接** | "不只是工具，是搭档" |

### 系统能力（HOW — 怎么实现产品目标）

| 能力 | 为什么 OpenCode 不做 | Muse 怎么做 |
|------|-------------------|-----------|
| **认知记忆** | OpenCode 是"工具"不需要记住用户 | Memory SQLite + MCP 暴露给 AI |
| **身份人格** | OpenCode 没有"性格" | identity.json → AGENTS.md + Skill |
| **多通道** | OpenCode 只有终端 | Telegram + Web + 未来硬件 |
| **Agent 家庭** | OpenCode 单用户工具 | 多 Agent 实例 + 共享记忆 |

**判断标准**: 做产品决策时看"产品目标"，做技术决策时看"系统能力"。其余的，站在 OpenCode 的肩膀上。

## 第六原则：健康自知与受控成长

> Agent 不只是会做事，还要知道自己状态怎么样。
> 不是"我们知道系统挂没挂"，而是"她知道自己哪里不舒服、要不要告诉 Later、先自己处理还是求助"。

### 6.1 三层职责划分

```
身体器官提供体征 → 小脑守护与本能 → 大脑解读与成长
```

| 层 | 职责 | 边界 |
|----|------|------|
| **身体** (Engine/Memory/Registry/Adapters) | 提供客观事实信号 | 被动汇报，不判断 |
| **小脑** (Cerebellum/Pulse) | 采集、定时检查、趋势检测、低风险自愈 | 管体征与本能，不替她思考 |
| **大脑** (AI/LLM) | 解读报告、判断行动、提出修复、规划成长 | 管理解与成长 |

**判断口诀**: "小脑管体征与本能，大脑管理解与成长。"

### 6.2 自愈 ≠ 成长

| | 自愈 (Healing) | 成长 (Growth) |
|--|---------------|--------------|
| 目的 | 恢复到稳定状态 | 获得新能力、优化自己 |
| 例子 | 重启服务、重连、清理 session | 新增 Skill、新 check 脚本、新 MCP 工具 |
| 风险 | 低 | 中到高 |
| 谁做 | 小脑可自动 | 大脑提议，人类审核 |

如果不区分，"自愈"会被无限扩张成"自己改自己一切"。

### 6.3 自愈安全分级

| 级别 | 风险 | 执行方式 | 例子 |
|------|------|---------|------|
| A 类 | 低 — 基础设施 | 小脑自动执行 | 重启、重连、刷新 |
| B 类 | 中 — 数据/配置 | 生成 patch，等审核 | 更新路由映射表、修改配置 |
| C 类 | 高 — 认知/规则 | 只能提建议 | 修改 AGENTS.md、调整人格、改核心逻辑 |

### 6.4 四句话检验

设计健康相关功能时，问自己：

1. **她知道自己现在状态怎么样吗？** (System Health)
2. **她知道自己会什么、不会什么吗？** (Self Model Health)
3. **她知道先自己处理，还是告诉 Later 吗？** (通知策略)
4. **她知道怎样在失败中成长，而不是只会报错吗？** (Life Health)

## 第七原则：Agent-First，工程是手脚

> 底层是工程，但以 Agent 的思路设计整个系统。用户面对的永远是 Agent，不是命令行。

### 7.1 思维转换

```
❌ 工程思维: 用户需要管理成员 → 设计 CLI 命令 → add-member / remove-member
✅ Agent 思维: 用户对 nvwa 说 "帮我加个会画画的 Muse" → nvwa 理解 → nvwa 执行
```

| 工程思维 | Agent 思维 |
|---------|-----------|
| 写 CLI 让用户敲命令 | Agent 理解自然语言后调用能力 |
| 写文档教用户配置 | Agent 引导用户完成配置 |
| 写迁移脚本让用户跑 | Agent 检测到需要迁移后自动处理 |
| 用户去查日志排错 | Agent 自己诊断并报告 |

### 7.2 分层原则

```
用户层:  自然语言对话（"帮我搬数据"、"加个新成员"）
Agent层: nvwa 理解意图 → 选择工具 → 执行 → 报告
工程层:  init.mjs / config-loader.mjs / migrate.mjs（Agent 的手脚）
```

- **工程层**为 Agent 提供能力，不直接面向用户
- **CLI** 仅用于开发调试和 Agent 无法启动时的应急操作
- **文档**写给 Agent 看（AGENTS.md），不是写给用户背命令用

### 7.3 检验口诀

设计新功能时问自己：**"用户需要知道这个命令吗？还是 nvwa 帮他做就行？"**

## 第八原则：AI 就是能力本身

> **T39-1.6 教训**: arch 被要求给 nvwa（AI Agent）添加"工作流创建能力"，结果设计了一套 NLP 解析器 + 模板引擎。但 nvwa 本身就是 AI——它天然能理解自然语言、生成 JSON。为 AI 已有的能力构建冗余模块，是最典型的过度工程化。

### 8.1 核心判断

```
❌ 为 AI 构建: NLP 解析器、模板引擎、意图分类器
✅ 给 AI 提供: Skill 文档、示例模板、约束规则
```

| 场景 | 传统做法（❌） | Agent 做法（✅） |
|------|-------------|---------------|
| 让 nvwa 会建工作流 | NLP 解析器 + 模板引擎 | Skill 文档 + 示例 JSON |
| 让 arch 理解架构 | 架构知识数据库 | 指向文档路径，AI 自己 read |
| 让 coder 遵守规范 | 代码生成模板 | AGENTS.md 写明规范 |

### 8.2 Prompt 是接口

对 AI Agent，AGENTS.md / system prompt 就是最好的"配置"和"技能"：

- 教 AI 新能力 → 写 Skill 文档（.md）
- 给 AI 约束 → 写到 AGENTS.md
- 给 AI 上下文 → 指向文档路径让它 read

**判断口诀**: "写代码前先想——写个 Skill 文档够不够？"

## 第九原则：工作流驱动协作

> 多个 Muse 成员协作时，通过工作流引擎而不是自由对话来驱动。

### 9.1 每个节点要素明确

工作流节点必须声明七要素：
1. **目标** — 这个节点做什么
2. **步骤** — 怎么做
3. **约束** — 不能做什么
4. **工具权限** — 能用什么工具
5. **退出条件** — 什么算完成
6. **状态流转** — 调什么 transition
7. **执行模式** — 自主执行 / 等待用户

### 9.2 自检保障

- **Pre-check**: 输入 artifact 不存在 → 暂停 + 通知用户
- **Post-check**: 输出 artifact 不存在 → 拒绝流转
- 不允许节点在没有产出物的情况下声称完成

### 9.3 上下文传递

- `input.artifacts` — 声明依赖的前序产出物
- `read_first` — 声明前置阅读的参考文档
- artifact 目录自动注入到 prompt — AI 知道去哪里找文件
