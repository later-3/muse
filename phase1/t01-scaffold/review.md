# T01 项目脚手架评审意见（基于 `phase1/t01-scaffold/README.md`）

总体结论：**方案方向正确、可落地，适合作为 Phase 1 基线**。但作为后续 T02-T09 的模板，建议在配置治理、运行时边界与风险前置控制上再补齐一层“工程化护栏”。

## 1) 架构合理性

### 评价
- 分层结构 `core / adapters / web / daemon / data` 清晰，职责边界符合“领域核心 + 适配层”思路。
- `muse/` 子目录承载业务源码是合理决策，避免与 `phase1/`、`assistant-prep/` 等工程文档目录耦合。
- `index.mjs` 作为装配入口，便于 T03-T08 分阶段接入，符合 MVP 递进式交付。

### 改进建议
- 建议尽早定义统一生命周期接口（例如 `start() / stop() / health()`），避免后续在入口层出现硬编码编排。
- 建议明确模块依赖方向：`adapters/web/daemon -> orchestrator -> engine/memory/identity`，禁止反向依赖，减少后期重构成本。
- 建议补一张轻量运行时拓扑图（启动顺序 + 失败传播路径），提升联调阶段可维护性。

## 2) 技术选型

### 评价
- Node.js 20 + ESM + `node:test` 选型务实，学习/维护成本低，适合 Phase 1 快速迭代。
- `telegraf` 作为 Telegram SDK 主流成熟，社区活跃度和文档都可接受。
- `better-sqlite3` 对单机本地记忆场景很契合（部署简单、性能稳定、事务语义明确）。

### 关注点与建议
- `better-sqlite3` 是同步 API，需明确并发假设（MVP 阶段可接受）；建议在文档增加“触发迁移条件”（如并发量/响应时延阈值）。
- `dotenv` 可继续使用，但 Node 20 已支持 `--env-file`，本地开发脚本可考虑双模式兼容，降低工具依赖耦合。
- 日志先用 console 封装可行，但建议统一结构字段（time/level/module/requestId），为 T09 问题排查预留观测性基础。

## 3) 配置系统

### 评价
- `config.mjs` 集中管理且有默认值，满足“先跑起来”的目标。
- 配置域拆分（telegram/engine/memory/web/identity/daemon）合理，便于后续模块直连使用。

### 主要缺口（建议在 T01 即补）
- 缺少启动时强校验（必填项、端口范围、路径合法性、数值下界）。
- `TELEGRAM_BOT_TOKEN` 为空时仍可启动，存在延迟失败问题；建议 fail-fast。
- `TELEGRAM_ALLOWED_USERS` 仅 split/filter，建议转为 number 并去重，防止授权判断隐患。
- 建议新增 `validateConfig()` 并在 `main()` 启动最前执行；校验失败直接退出并打印可读错误。

## 4) 风险识别

### 已覆盖风险（文档中已有）
- `better-sqlite3` 编译依赖风险。
- `telegraf` 大版本行为变化风险。
- `muse/` 子目录布局的组织合理性争议。

### 建议补充风险（当前遗漏）
- **安全风险**：`.env` 泄露、Bot Token 泄露、白名单配置错误导致未授权访问。
- **稳定性风险**：OpenCode 不可达或慢响应导致级联阻塞。
- **运行风险**：launchd 重启风暴（异常快速退出循环）与告警缺失。
- **数据风险**：SQLite 损坏/锁竞争，建议在 memory 层初始化 WAL 和备份策略。

## 5) 可行性

### 结论
- 从工程可执行角度看，**可以直接开始开发**，并且 T01 已具备作为后续任务基座的必要条件。

### 上线前最小补强清单（建议）
1. 增加配置校验与 fail-fast。
2. 在 DoD 中将“git commit”替换为“可验证产物 + 自动化检查通过（启动验证/配置测试）”。
3. 明确生命周期与依赖方向约束，降低 T05 以后集成复杂度。
4. 为 Telegram 入口预留最小安全基线（白名单中间件 + 简单限流）。

## 评审结论（执行建议）

- **READY WITH ACTIONS**：方案可进入实现，但建议先完成上述 4 项补强，再并行推进 T02/T03。
- 若按当前文档原样推进，也能完成 MVP；但后续在联调阶段可能集中暴露配置和稳定性问题，修复成本会更高。
