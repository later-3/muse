## 项目背景

Muse 是基于 OpenCode 的 AI 灵魂伴侣框架（不是助手）。核心架构:
- **大脑** = OpenCode serve (推理、44 个工具、代码生成)
- **灵魂** = Muse Agent (记忆、身份、主动性)
- **小脑** = Cerebellum (守护、监控、定时任务)

Phase 1 已完成 T01-T07。T08 = 小脑实现，分 2 部分:
- T08-1: 核心守护 (心跳 + 进程管理 + 会话清理)
- **T08-2: 自我成长引擎 (本次审核对象)**

## 核心设计哲学

1. **AI 做认知决策，工程层做安全保障** (PHILOSOPHY.md 第一原则)
2. **站在 OpenCode 的肩膀上** — 用 OpenCode 的工具修改自己的代码
3. **Plugins > Forks** — 不直接改 OpenCode 源码
4. **Sisyphus 式失败恢复** — STOP → REVERT → DOCUMENT → CONSULT

## 审核结果

### 1. 总体评价

⚠️ **方向是对的，但当前 `T08` 目录里的文档层次和阶段边界还没有收拢。**

结合 `phase1/README.md`、`ARCHITECTURE.md`、`PHILOSOPHY.md`、`phase2/README.md` 和当前 `T01-T07` 的真实进展，我的判断是：

1. `T08-1 小脑守护` 是 **Phase 1 必需项**，而且重要性已经上升
   - 因为 `T07` 暴露出的真实问题，已经不是“再加一个功能”能解决的，而是系统启动、降级、恢复和诊断能力需要补齐
2. `T08-2 自我成长引擎` 的产品想象力很强，也和 Muse 长期目标一致
   - 但它现在更像 **Phase 1.5 / Phase 2 预研方案**
   - 不适合和 `T08-1` 混在同一个交付层级里一起承诺
3. 当前最大的文档问题不是单点技术，而是：
   - `README.md` 在写 `T08-1`
   - `review-prompt.md` 在审 `T08-2`
   - 目录里没有 `context.md`
   - 这会直接导致开发和审核对象错位

所以结论不是“方案不行”，而是：**T08 需要先分层，再开发。**

### 2. 主要优点

✅ **以下设计是好的，而且和 Muse 的整体路线一致。**

1. **两级守护思路正确**
   - `launchd → Cerebellum → OpenCode serve`
   - 这和 `phase1/README.md`、`ARCHITECTURE.md` 的“大脑/小脑”结构一致
   - 也符合 `PHILOSOPHY.md` 里“工程管生死”的原则

2. **小脑不复用 Engine 类，而是用轻量 REST 探针**
   - 这个决策是对的
   - 小脑应该尽量简单、可恢复、少依赖
   - 否则小脑自己又变成一个重客户端，故障面会扩大

3. **会话清理纳入小脑职责是合理的**
   - 结合 `T05` 当前的 session 使用方式，以及 `T07` 暴露出的上下文膨胀问题，`>24h session` 清理是合理且必要的

4. **自我成长引擎的哲学方向是对的**
   - “AI 感知能力缺口，工程提供安全的成长轨道”这件事和 Muse 的长期目标是同向的
   - 它不是纯自动 patch 炫技，而是在尝试把“成长”变成系统能力

5. **Growth Journal 放进可视化轨迹的想法是有价值的**
   - 这和 T07 的“成长系统”长期方向能接上
   - 也能增强 Muse 不只是工具、而是会演化的存在感

### 3. 关键问题

❌ **下面这些是必须先明确或修正的。**

#### 问题 1：T08 的文档对象已经错位

当前目录里存在明显冲突：

1. [README.md](/home/user/Code/assistant-agent/phase1/t08-daemon/README.md) 定义的是 `T08-1 小脑核心`
2. [review-prompt.md](/home/user/Code/assistant-agent/phase1/t08-daemon/review-prompt.md) 审的却是 `T08-2 自我成长引擎`
3. 目录里没有 `context.md`
4. 但 `review-prompt.md` 又写了“请阅读 `self-growth-spec.md`”

这不是排版小问题，而是会直接导致：

1. 开发人员不知道当前 T08 的正式交付对象是什么
2. 审核意见会混杂 Phase 1 MVP 和未来演进
3. T09 联调时无法判断“哪些没做是缺陷，哪些本来就不在本阶段”

**建议：**

1. 把 `T08-1` 和 `T08-2` 明确拆开
2. 当前 `phase1/README.md` 的 T08 只应对应 `T08-1 小脑守护`
3. `T08-2` 建议改名为：
   - `T08-2 自我成长预研`
   - 或直接移到 `phase2/` / `phaseX-labs/`

#### 问题 2：T08-1 的健康探针和现有 Engine 真实行为没有完全对齐

[README.md](/home/user/Code/assistant-agent/phase1/t08-daemon/README.md#L55) 和 [README.md](/home/user/Code/assistant-agent/phase1/t08-daemon/README.md#L279) 里把健康检查写成 `GET /provider`，但结合当前 `EXPERIENCE.md`：

1. `BUG-004` 已明确：OpenCode v1.2.24 优先健康端点是 `/global/health`
2. `Engine.health()` 现在就是优先 `/global/health`，再降级 `/provider`

如果小脑仍只打 `/provider`，就会出现：

1. 小脑判断“大脑死了”
2. 实际只是探针过时
3. 然后误触发重启

这在守护进程里是高风险设计。

**建议：**

1. 小脑健康检查逻辑必须和当前 `Engine` 的兼容策略保持一致
2. 最好抽成一个共享的“只读探针工具”，而不是完全复制旧逻辑

#### 问题 3：T08-1 当前启动关系和 T07 的现实问题还没完全打通

从整体路线看，`T08` 现在最重要的价值不是“自动重启很酷”，而是要解决：

1. 大脑挂了以后系统是否还能恢复
2. Web 是否还能作为诊断入口
3. 适配器是否能在部分模块异常时继续提供有限服务

但当前 T08 文档更多强调：

1. 心跳
2. 重启
3. 清 session

而没有把“**部分故障时，系统如何降级可用**”写成一等目标。

结合 `T07` 的现状，这里建议补一个明确目标：

1. Web 应尽量先于依赖模块启动，成为故障诊断入口
2. 小脑提供给 Web 的状态不应只是“活/死”，还应包括：
   - 最近重启时间
   - 连续失败次数
   - 最近一次探针失败原因
   - 当前是否处于 attach / owned 模式

#### 问题 4：T08-2 自我成长方案过早跨越到了“自动改代码并热安装”

`self-growth-spec.md` 的想法有吸引力，但它当前一口气承诺了太多高风险能力：

1. 感知缺口
2. 自动研究
3. 自动改代码
4. 自动建分支
5. 自动跑测试
6. 自动合并
7. 热加载
8. 给用户回复“我学会了”

这在 Muse 的长期路线里不是不能做，而是**不适合当前阶段直接作为 T08 交付承诺**。原因有 4 个：

1. 当前 `T01-T07` 还在补系统一体化，不是自动演化平台阶段
2. `T07` 和 `T08-1` 还没有先把基础诊断、降级、恢复坐实
3. `TelegramAdapter` 目前也还不是完整多模态输入框架
4. “热加载” 对现有 Node.js ESM 模块和 adapter 生命周期管理来说还远未明确

这类方案更适合先拆成更小的阶段：

1. `Phase A`：只做 CapabilityGap 记录与可视化，不改代码
2. `Phase B`：人工触发学习任务，生成研究方案和 patch 提议
3. `Phase C`：受控批准后自动修改和测试
4. `Phase D`：再讨论自动安装与热加载

#### 问题 5：T08-2 在哲学上方向对，但在工程边界上还不够收敛

`PHILOSOPHY.md` 的核心是：

1. AI 做认知决策
2. 工程层负责安全、生命周期、资源、数据一致性

而 `self-growth-spec.md` 现在把“自动成长”推进到了：

1. 自动 branch
2. 自动 commit
3. 自动 merge main
4. 自动热加载

这会让工程层边界变得过薄。对于一个会自己改代码的系统，真正危险的不是“会不会写出 bug”，而是：

1. 有没有可能在错误前提下做出正确形式但错误方向的修改
2. 有没有可能通过测试但破坏未覆盖行为
3. 有没有可能进入反复修自己、反复失败的资源黑洞

所以这里最大的问题不是“AI 能不能改代码”，而是**自动安装门槛过低**。

文档里虽然写了：

1. 超过 200 行需审批
2. 3 文件以上需审批
3. 核心模块需审批

但这还不够。

**建议至少再加 4 条硬限制：**

1. 默认不允许自动 merge 到 `main`
2. 默认不允许自动热加载生产实例
3. 默认只允许生成 patch / branch / 测试结果，等待人工批准
4. 单日成长尝试次数和总 token / 总时长必须有硬上限

### 4. 建议补充

⚠️ **这些不是当前阻塞，但补上后方案会稳很多。**

1. **给 T08 建立清晰分层**

建议目录和任务口径改成：

1. `T08-1 守护小脑`
2. `T08-2 成长引擎预研`

当前 Phase 1 只承诺 `T08-1`，`T08-2` 作为附录或下一阶段设计稿。

2. **补一个真正的 `context.md`**

当前没有 `context.md`，但这是守护与成长模块最需要上下文的地方。至少应补：

1. 当前 `Engine` 真实 API 行为
2. 当前 `Telegram/Web` 真实能力边界
3. `T07` 暴露出来的启动与诊断问题
4. `Phase 2` 对原生 agent 化的要求

3. **把 T08-1 的“诊断可观测性”升到一等公民**

建议小脑对外暴露的信息不只 `ok/fail`，还包括：

1. 最近 10 次心跳结果
2. 最近一次重启时间
3. 最近一次失败原因
4. 当前附着的是外部大脑还是自管大脑
5. 当前 session GC 最近一次执行结果

这样 T07 才真正能成为“驾驶舱”。

4. **把 T08-2 的最小可行版本改成“记录 + 建议”，不是“自动安装”**

如果你刚加的需求是希望 Muse 开始具备“成长感”，我建议第一版先收敛到：

1. 能识别 CapabilityGap
2. 能记录到 Growth Journal
3. 能在 Web 里展示成长轨迹
4. 能生成一份学习方案或 patch 提议

先让“她会意识到自己不会什么”成立，再谈“她会安全地学会”。

5. **与 Phase 2 衔接建议**

`T08-2` 真正合适的下一阶段衔接点是：

1. 用 MCP 工具而不是内部私有逻辑暴露成长能力
2. 把成长任务作为 Agent 能力，而不是 daemon 私有魔法
3. 让小脑负责触发与守护，成长决策交给更原生的 agent 流程

### 5. 结论

**结论：**

1. `T08-1 小脑守护`：**可以进入开发，但需要先修正文档与现有系统现实的对齐问题**
2. `T08-2 自我成长引擎`：**方向成立，但不建议作为当前 Phase 1 正式交付承诺，建议下沉为预研/下一阶段设计**

**T08-1 必须先修正的点：**

1. 小脑探针和当前 OpenCode 健康检查现实对齐（`/global/health` 优先）
2. 把 T08 和 T07 的诊断入口、降级可用性明确联动
3. 修正文档对象错位：README / review-prompt / self-growth-spec 的层次关系

**T08-2 建议的通过条件：**

1. 不承诺自动 merge main
2. 不承诺自动热加载生产实例
3. 先降级为“缺口感知 + 成长日志 + 人工审批的 patch 提议”
4. 明确它属于 `Phase 1.5 / Phase 2` 演进，而不是当前 `T08` 的 MVP 核心

一句话判断：

**Muse 当前最需要的是一个可靠的小脑，不是一个马上会自己改代码的小脑。成长方向值得保留，但守护优先于进化。**
