## 评审维度

请从以下 6 个维度逐项评审：

### 1. 核心流程完整性

- handleMessage 的 6 步流程是否覆盖了 Phase 1 README 定义的编排需求？
- 是否有遗漏的关键步骤？
- 流程中的数据流是否清晰、无歧义？

### 2. 接口边界正确性

- T05 调用的上游 API (Identity/Memory/Engine) 是否和 context.md 描述的接口签名一致？
- T05 对外暴露的 `handleMessage` 返回值是否满足 T06/T07 的需求？
- Orchestrator 是否越权做了不该做的事（如管理 session 清理、管理子模块生命周期）？

### 3. 意图分类设计

- 规则 + 关键词的 Phase 1 方案是否够用？
- 是否有明显的误分类场景没有考虑到？
- 分类结果对上层的影响是否可控？（误分类最坏后果是什么？）

### 4. 记忆注入策略

- prompt 组装模板是否合理？token 预算是否安全？
- 语义记忆的单关键词搜索在 Phase 1 是否够用？
- 情景摘要注入的 3 天窗口是否合理？
- 偏好提取的正则策略是否有明显误触发风险？

### 5. 错误处理与健壮性

- 后处理异步失败是否正确处理了（不阻塞返回 + log）？
- Engine 超时和网络错误是否有处理？（还是完全透传 Engine 内部的错误处理？）
- 如果 Identity/Memory 某个模块 health 为 false，handleMessage 是否仍然可用？
- 是否有需要但缺少的防御性检查？

### 6. 测试覆盖度

- 24 项测试是否覆盖了所有公开 API 和关键内部逻辑？
- Mock 策略是否合理？是否有"测试只测了 mock"的风险？
- 是否有遗漏的边界场景？
- 测试是否和实际的 T01-T04 接口一致（不会出现 API mismatch）？

## 评审重点提醒

1. **不要过度设计** — T05 是 Phase 1 MVP，规则分类、LIKE 搜索、正则提取都是可接受的简化
2. **关注 T06/T07 消费者视角** — handleMessage 的 API 设计要对下游友好
3. **关注 T02/T03/T04 的真实接口** — 方案中调用的方法必须和现有代码完全一致
4. **Orchestrator 不应该有自己的状态** — 它是无状态粘合层，不持有 DB、不持有进程
5. **后处理失败必须不阻塞** — 用户体验优先，记忆写失败可以降级

# T05 编排层 — 评审报告

## 总体评价

方案主线是对的，T05 作为 `Identity → Memory → Engine` 的无状态粘合层，职责边界整体清楚，`handleMessage()` 也基本覆盖了 Phase 1 需要的主链路。  
但当前版本**还不建议直接开始开发**，因为有 `3` 个会影响真实效果的关键问题：中文关键词提取策略几乎不可用、Memory/Identity 异常时没有降级路径、后处理失败粒度过粗。

## 逐项评审

### 1. 核心流程完整性 [⚠️]

主流程的 `5` 个阶段是完整的：

1. 意图分类
2. session 解析
3. prompt 构建
4. 引擎调用
5. 异步后处理

从 Phase 1 README 的验收目标来看，这条链已经覆盖了：

- 人格注入
- 记忆注入
- 模型路由
- 记忆回写

这部分没有明显缺步骤。

问题主要有 `2` 个：

1. 文档里的主流程写了“6 步”，但实现实际上是 `5` 段，表述有点不一致，不过这是文档层小问题。
2. `#buildPrompt()` 和 `#postProcess()` 都假设上游模块正常可用，没有定义“读不到记忆时如何退化”的路径。流程是完整的，但还不够健壮。

### 2. 接口边界正确性 [✅]

这部分整体是成立的。

- 对 Identity 的调用只用了 `buildSystemPrompt()` / `health()`，符合 context.md
- 对 Memory 的调用只用了公开 CRUD 接口，也和 context.md 一致
- 对 Engine 的主调用使用 `sendAndWait()`，这正是 T03 暴露给 T05 的主要入口
- 对外返回 `{ text, sessionId, model, intent }`，满足 T06/T07 的消费需求
- Orchestrator 没有越权去管理引擎进程、数据库连接或 session 清理，这点做得对

唯一的小瑕疵是 README 的接口约定里提到了 `Identity.data` 可用于欢迎语，但当前核心实现并没有用到它。不过这不构成问题，只是文档里有一点“预留但未使用”。

### 3. 意图分类设计 [⚠️]

Phase 1 用规则 + 关键词分类是合理的，这本身不是问题。

优点：

- 零额外 LLM 调用
- 零成本、零延迟
- 行为可预测、易测试

但当前规则有 `2` 个明显风险：

1. 正则里有一些写法实际上是字符类，不是词组。  
例如：
```js
/写[一个]*(?:代码|函数|脚本|程序|类|组件)/
```
这里 `[一个]*` 不是“可选的‘一个’”，而是“匹配任意多个‘一’或‘个’字符”。  
同类问题还出现在：
```js
/写[一篇]*(?:文章|文档|报告|方案|总结)/
```
虽然多数情况下还能误打误撞工作，但规则语义并不精确，后续维护容易误判。

2. 误分类的兜底语义没有写清楚。  
当前所有未命中的都落到 `light`，这意味着最坏情况是：
- 本该走 `heavyModel` 的复杂任务被送到轻量模型
- 输出质量下降，但功能仍可完成

这个后果在 Phase 1 是可以接受的，但文档里建议明确写出来：  
“误分类的最坏后果是回复质量下降，不是流程失败”，这样设计边界会更清楚。

### 4. 记忆注入策略 [❌]

这是当前方案最大的问题点。

#### 4.1 中文关键词提取策略几乎不可用

当前实现：

```js
const cleaned = text.replace(..., ' ')
const words = cleaned.split(/\s+/).filter(w => w.length >= 2)
return words.slice(0, 3)
```

这对英文或“中文里恰好有空格”的输入能工作，但 Muse 的主要语言环境是中文。  
对于常见中文输入，例如：

```text
我昨天说过我喜欢喝拿铁
帮我写JavaScript函数
你还记得我喜欢什么吗
```

由于中文句子通常**没有空格**，这个方法往往只会得到整句，甚至根本提不出可用关键词。  
而你后面又只做：

```js
memory.searchMemories(keywords[0])
```

这意味着语义记忆检索在中文主场景下会非常弱，直接影响“她记得昨天的内容”这个 Phase 1 验收目标。

这不是优化项，而是 blocker。  
至少要改成更适合中文的 Phase 1 简化方案，例如：

- 对中文按 `2-4` 字滑窗生成候选词
- 或针对“喜欢/偏好/叫/语言/编辑器”等偏好类词做规则抽取
- 或直接先从全文里挑出匹配 memory key/value 的短片段

#### 4.2 记忆检索只用第一个关键词，命中率过低

即使关键词提取本身能工作，当前仍然只查：

```js
memory.searchMemories(keywords[0])
```

这会让后两个关键词完全失效。  
Phase 1 不一定要上复杂 OR 查询，但建议至少串行查前 `2-3` 个关键词后去重，否则“只取第一个词”过于脆弱。

#### 4.3 偏好提取正则有误触发和截断风险

例如：

```js
/我喜欢(?:用|使用)?\s*(\S+)/
```

面对：

```text
我喜欢用 VSCode 写 TypeScript
```

提取出来的很可能是整段 `VSCode写TypeScript` 或只到第一个空白前，行为不稳定。  
Phase 1 用正则没问题，但建议把提取目标定义得更窄一点，例如只抽短值，或者先按标点截断。

### 5. 错误处理与健壮性 [❌]

当前方案的方向是正确的：后处理失败不阻塞返回，这一点符合用户体验优先。

但具体实现还缺 `3` 个关键防御：

#### 5.1 Memory / Identity 异常时没有降级路径

`#buildPrompt()` 里直接调用：

```js
this.#identity.buildSystemPrompt()
this.#memory.searchMemories(...)
this.#memory.getRecentSummaries(3)
```

如果 Memory 当前不可用，`T04` 的行为是直接抛：

```text
Memory not started
```

那 `handleMessage()` 就会整体失败。  
但从产品角度，Memory 挂了时用户其实仍然应该能得到“无记忆模式”的回复；Identity 挂了时，最差也可以退化成一个默认 system prompt。

因此这里应该明确：

- Memory 失败 → 记录日志，退化为空记忆
- Identity 失败 → 使用最小默认 persona 或至少给出明确错误语义
- Engine 失败 → 直接透传，因为它是核心依赖

现在文档没有定义这层降级策略，这是 blocker。

#### 5.2 后处理是整段 `.catch()`，失败粒度过粗

当前：

```js
this.#postProcess(...).catch(...)
```

但 `#postProcess()` 内部是串行做：

1. `addEpisode(user)`
2. `addEpisode(assistant)`
3. `extractPreferences()`

如果第 `1` 步抛错，后两步都不会执行。  
这虽然满足了“不阻塞返回”，但不满足“尽量保留可写入信息”。建议至少把这三步各自 try/catch 或按小段隔离，避免一处失败导致后续全部跳过。

#### 5.3 缺少输入校验

`handleMessage(text, context)` 没有检查：

- `text` 是否为字符串
- 是否为空串
- 是否全空白

这类检查很轻，但能避免适配器层漏处理时把脏输入一路传给 Engine。

### 6. 测试覆盖度 [⚠️]

测试矩阵的覆盖面不错，`24` 项已经把主路径、模型路由、prompt 组装、后处理和 health 聚合都列出来了。

但还缺 `3` 类关键场景：

1. **Memory 不可用时的降级测试**  
这正是当前方案最薄弱的地方，但测试里没有覆盖。

2. **中文连续文本的关键词提取测试**  
你列了：
```text
"帮我写JavaScript函数" → 提取出有效关键词
```
但按当前实现，这条测试大概率不会稳定成立。说明测试目标和设计之间已经有偏差。

3. **后处理部分失败但部分成功的测试**  
现在只测“后处理整体失败不阻塞”，没有测“某一步失败时，其他步骤仍然尽量执行”。

另外，全部使用 mock 是合理的，但要确保 mock 的接口和 T02/T03/T04 当前真实接口完全一致。就目前文档看，基本一致，没有明显 mismatch。

## 必须修正 (Blockers)

1. 重做中文关键词提取策略。当前按空格切词 + 只取第一个关键词的做法，会让中文主场景下的语义记忆检索效果非常差，不足以支撑 Phase 1 的“记得上下文”目标。
2. 为 `Identity` / `Memory` 增加降级策略。至少要定义：Memory 失败时退化为空记忆而不是整条链路失败。
3. 把后处理从“整段 catch”改成“分步骤容错”，避免 `addEpisode(user)` 一次失败导致 assistant 记录和偏好提取都丢掉。

## 建议改进 (Suggestions)

1. 修正意图分类正则里把词组写成字符类的问题，例如 `[一个]*`、`[一篇]*`。
2. 给 `handleMessage()` 增加输入校验，拒绝空串/非字符串输入。
3. 语义记忆检索不要只查 `keywords[0]`，至少合并前 `2-3` 个关键词的结果后去重。
4. 在 README 里明确说明误分类的最坏后果是“模型选弱，质量下降”，不是系统错误。
5. 偏好提取规则建议加上值长度限制或标点截断，减少误触发。
6. 补测试：中文关键词提取、Memory 降级、后处理部分失败。

## 确认通过的部分

1. Orchestrator 作为无状态粘合层的定位是正确的，没有越权管理子模块生命周期。
2. `handleMessage()` 作为 T06/T07 的统一入口，这个 API 设计是对的。
3. 返回值 `{ text, sessionId, model, intent }` 对下游足够友好，调试信息也足够。
4. `sendAndWait()` 作为主引擎入口、Orchestrator 不重复实现重试逻辑，这个边界划分是合理的。
5. “后处理异步不阻塞返回”这个方向是正确的，符合用户体验优先。
6. Phase 1 不做额外 LLM 分类、不做向量检索、不做摘要生成，这些取舍整体克制，没有过度设计。

# T05 编排层 — 补充审核（基于 assistant-agent 项目真实文档与代码）

## 总体评价

这版补充审核只基于 `assistant-agent` 项目内已经存在的文档和代码来判断，不参考工作区其他项目。  
结合当前 `T01-T04` 的真实实现后，`T05` 方案依然是可行的，但有 `2` 个新的高优先级偏差需要写进文档，否则实现出来后很容易“代码没错，但效果比文档预期弱很多”。

## 逐项补充

### 1. Prompt 注入方式和项目现有 Engine 能力之间有现实落差 [⚠️]

从当前项目真实代码看：

- `Identity.buildSystemPrompt()` 会返回一段结构化人格文本
- 但 `Engine.sendAndWait(sessionId, text, opts)` 对外只接收一个 `text` 字符串
- `Engine.sendMessageAsync()` 最终发送给 OpenCode 的是：

```js
{ parts: [{ type: 'text', text }] }
```

这意味着 T05 文档里的：

```text
[System]
{identity.buildSystemPrompt()}

[Memory Context]
...

[User]
{用户原始消息}
```

在当前项目中并不是模型级别的真正 `system role + user role` 输入，而是**拼成一整段普通文本**发给引擎。  
这不会让方案失效，但会直接影响人格约束和记忆注入的稳定性，也就是：

1. “有人格的回复”会弱于真正 system 通道
2. “记得昨天的内容”会更依赖提示词措辞而不是输入层级

因此建议在 T05 README 里明确：

- 当前是“拼接式伪 system prompt”
- 这是 Phase 1 的现实折中，不是最终形态
- Phase 2 如果要提升稳定性，需要更强的 prompt 结构化输入能力

如果不写清楚，后面很容易把“有 system prompt 文本”误当成“已经有真正 system 注入能力”。

### 2. T05 文档和 Phase 1 总图、T04 真实能力在“关键词检索数量”上不一致 [❌]

`phase1/README.md` 里，T05 核心流程写的是：

```text
检索相关语义记忆 (top-5 关键词匹配)
```

但 T05 README 当前写法是：

```js
const keywords = this.#extractKeywords(userText)
const semanticHits = keywords.length > 0
  ? this.#memory.searchMemories(keywords[0])
  : []
```

再结合当前 `T04 Memory` 的真实接口：

- `searchMemories(keyword)` 只支持**单关键词 LIKE**
- 没有多关键词 OR 查询，也没有批量检索接口

所以当前项目的真实能力其实是：

1. 总图写的是 `top-5`
2. T05 方案实现的是 `top-1`
3. T04 实际接口也只天然支持 `top-1` 风格调用

这不是措辞问题，而是一个真实的项目内不一致。  
如果不修正文档，后面很容易出现“按总图理解是 top-5，但实际实现只有 top-1”的预期偏差。

建议二选一：

1. 把 `phase1/README.md` 和 T05 README 统一改成“Phase 1 先做单关键词或 top-3 串行检索”
2. 或者在 T05 里明确会串行调用 `searchMemories()` 多次后合并去重

当前写法下，我更建议先统一成“单关键词简化检索”，因为这才符合项目当前真实代码基础。

### 3. 基于当前 T02/T04 的真实行为，T05 必须把降级策略写得更明确 [⚠️]

从项目当前代码看：

- `Memory` 在未启动或 stop 后调用公开方法，会直接抛：
  ```text
  Memory not started — call start() first
  ```
- `Identity.buildSystemPrompt()` 也依赖内部已加载状态
- `Engine` 则是核心依赖，失败时理应直接让请求失败

因此 T05 不能只写“调用三模块公开 API”，还必须明确：

1. Memory 不可用时是否允许降级为空记忆模式
2. Identity 不可用时是否允许降级为最小默认 persona
3. 哪些错误应该直接失败，哪些应该记录日志后继续

按当前项目目标“先跑通核心循环”，我建议文档明确成：

- `Engine` 不可用：`handleMessage()` 失败
- `Memory` 不可用：降级为无记忆模式
- `Identity` 不可用：降级为最小默认 persona 或明确错误语义

你前一版评审里已经提到降级问题，但这里补充的是：**这不是抽象建议，而是由当前 `assistant-agent` 项目真实代码行为直接决定的。**

## 补充结论

基于 `assistant-agent` 当前真实文档和代码，T05 文档还需要再补清楚 `3` 件事：

1. 当前 prompt 注入不是模型原生 system role，而是拼接式文本
2. 当前语义记忆检索能力和总图里的 `top-5` 描述不一致
3. 当前上游模块的真实错误语义要求 T05 明确降级策略

# T05 编排层 — 开发结果补充审核（结合整体 Muse 需求）

## 总体判断

T05 当前实现方向是成立的，中文关键词提取、多关键词检索、降级策略、分步后处理这些关键点都已经明显优于最初方案。  
但站在 **Muse 整体目标** 来看，当前还有 `2` 个需要明确提出的问题：它们不一定都要在本阶段立刻彻底解决，但必须先在设计上承认并记录下来，否则后续 T06/T07/T08 集成时会反复踩坑。

## 发现的问题

### 1. 长 session 复用 + 每轮重新注入完整 prompt，会导致上下文污染和冗余膨胀

当前 T05 的处理方式是：

1. 复用 `context.sessionId`（如果适配器传入）
2. 每轮都重新拼接：
   - `identity.buildSystemPrompt()`
   - 语义记忆
   - 最近摘要
   - 用户原始消息
3. 再把这整段文本作为普通消息发给 T03 Engine

而当前项目里的 T03 Engine 并没有真正的 `system role` 注入通道，发送给 OpenCode 的仍然是：

```js
parts: [{ type: 'text', text }]
```

这意味着：

- 人格 prompt 不是系统层，而是普通消息内容
- 记忆摘要不是外部检索结果，而是被塞进 session 历史
- 同一个 session 越聊越长，会不断累积重复的人格说明和记忆上下文

这在短会话里未必马上出问题，但从 Muse “长期对话、有记忆、持续陪伴”的目标看，这个策略会带来 `3` 个真实风险：

1. 上下文越来越臃肿
2. 模型对真正用户新输入的关注度被稀释
3. 人格和记忆信息重复出现，反而降低 prompt 稳定性

### 2. session 生命周期和 T08 小脑的清理策略之间还没有真正打通

T05 目前把 `context.sessionId` 视为可直接复用的 session。  
但从 Phase 1 总图看，T08 小脑后面会做僵尸 session 清理。

这就带来一个现实问题：

- 上层适配器可能还保留着旧的 `sessionId`
- 但 OpenCode 侧 session 已经被小脑清掉
- 下一次 `handleMessage()` 如果仍直接用这个旧 `sessionId`，就会失败

当前 T05 还没有定义：

1. 如何检测 session 是否仍然有效
2. 如果 session 失效，是自动新建还是返回给上层处理
3. 适配器侧的 session 映射应如何和小脑清理策略协同

这不是单纯的 T05 局部问题，而是 T05 和 T08 的协作边界问题。

## 建议

### A. 本阶段建议先明确策略，不一定马上彻底实现

当前最重要的不是立刻把所有策略都做复杂，而是**先明确原则**。建议开发同学继续思考下面这几个方向：

1. **短 session + 外部长期记忆**
   - 每次对话或每个时间窗口使用较短 session
   - 长期记忆完全依赖 T04 检索注入
   - 优点：上下文不会无限膨胀
   - 缺点：每轮都更依赖外部 prompt 注入质量

2. **长 session + 轻量增量注入**
   - 第一次建 session 时注入完整 persona
   - 后续轮次只注入必要的记忆增量，不重复塞完整 prompt
   - 优点：更接近“持续会话”
   - 缺点：需要更细的 prompt 策略和 session 状态管理

3. **分层上下文策略**
   - persona：低频注入
   - 语义记忆：按需检索注入
   - 情景摘要：只在跨天/跨轮较久时注入
   - 当前用户消息：每轮必传
   - 这可能更符合 Muse 的长期演进方向

从 Muse 的长期目标看，我更倾向于第 `3` 种方向，但 Phase 1 未必要一次做完。

### B. 本阶段最少应补的设计约束

即便当前先不彻底重构 session 策略，也建议至少在文档中明确下面几点：

1. **当前实现是 Phase 1 的过渡方案**
   - 允许“复用 session + 拼接 prompt”
   - 但要明确它有上下文膨胀风险

2. **未来需要做 smarter context management**
   - 不是简单把更多记忆塞进 prompt
   - 而是根据会话长度、时间跨度、任务类型动态决定注入什么

3. **T05 与 T08 的 session 协同需要单独设计**
   - `sessionId` 不能被视为永久有效
   - 清理后如何恢复，是一个明确的后续设计点

### C. 可以在后续阶段优化解决，不必当前全部阻塞

下面这些问题我认为**不一定要求当前阶段全部解决**，但必须先记录为后续优化项：

1. persona 从“拼接式文本”升级到更强的 system-level 注入
2. session 失效自动恢复策略
3. 长 session 的上下文压缩 / context compaction
4. 情景摘要从“消息级注入”升级为“会话级摘要 + 分层摘要”
5. 基于任务类型动态控制记忆注入量，而不是固定模板

## 给开发人员的思考方向

这部分不应只当成“要改什么”，更重要的是继续想清楚：

1. Muse 到底更像“短任务助手”还是“持续陪伴型助手”？
2. 如果是持续陪伴型，哪些上下文应该长期驻留，哪些应该按需召回？
3. session 的存在价值是什么：承载短期工作记忆，还是承载长期人格与关系？
4. T04 Memory 和 T03 Engine 应该如何分工，才能既稳定又节省上下文？

这些问题现在先提出来，比急着拍板一个固定策略更重要。  
建议把它们作为 T05 后续优化记录，供 T06/T07/T08 和后续 Phase 2 一起收敛。
