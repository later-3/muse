## 核心教學內容提煉

### 1. 深度思考的代價問題

[Fact] Reasoning 模型的核心矛盾：想得越多，答案越好，但成本也越高。

> "深度思考的 token 越長，推理時間越長，API 費用越高。但並非所有問題都需要深度思考。"

一個 1+1=2 的問題如果也要思考 500 個 token，就是浪費。

### 2. 怎麼讓模型"少想"

[Fact] 幾種縮短 Reasoning 的方法：

- **Budget Forcing**：直接限制模型思考過程的最大 token 數。粗暴但有效。
- **早期終止**：當模型的思考過程出現"我已經確定了"類似的信號時，強制結束思考。
- **蒸餾短思考**：用長思考的結果去訓練一個短思考的模型。

### 3. 思考長度和準確率的關係

[Fact] 有趣的實驗發現：

- 簡單問題：思考太長反而可能**降低**準確率（想太多）
- 困難問題：思考越長準確率越高
- 存在一個"最佳思考長度"，取決於問題難度

### 4. 動態分配思考量

[Fact] 最理想的方案是讓模型自己判斷"這個問題需要想多久"：

- 簡單問題：快速回答，不要深度思考
- 複雜問題：允許充分思考
- 這需要一個"難度感知"機制，目前仍是開放問題

### 5. 實用策略

[Fact] 在 Agent 場景中，Reasoning 縮短尤其重要：

- Agent 每輪互動都會消耗 Context Window
- 如果每輪都做深度思考，Context 消耗速度會快很多
- 簡單的工具調用（讀文件、寫文件）不需要深度思考，但複雜的規劃和決策需要

## 設計動機與第一性原理

- **為什麼要縮短？** — Token = 錢 = 時間 = Context 消耗。Agent 場景中尤其致命。
- **核心 tradeoff** — 思考時間 vs 準確率。最佳點因問題而異。
- **與 LH25_07 的關係** — LH25_07 講 Reasoning 的能力，本講講 Reasoning 的成本控制。一個講加法，一個講減法。
