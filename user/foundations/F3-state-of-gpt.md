# F3: State of GPT — Karpathy

> **来源：** Andrej Karpathy Microsoft Build 演讲 (2023.05) · ~1h
> **一句话：** GPT 从一堆文本变成一个有用的 AI 助手，经历了哪些阶段。

---

## ⚡ 3 分钟速读版

```
一句话: LLM 训练 = 预训练(知识) → SFT(行为) → RLHF(偏好) → 推理优化(部署)
3 个关键概念: SFT 改行为不改知识 / RLHF 的 reward model / 推理时的采样策略
对 Muse 最重要的: 理解"LLM 的每个能力都有具体的训练来源"
面试必记: 严格区分"模型知道什么(预训练)"和"模型怎么表现(SFT+RLHF)"
```

---

## 🔬 核心原理（三栏表）

| 概念 | 能力来源 | 激活方式 | 类比（仅类比） |
|------|---------|---------|--------------|
| **SFT** | 人类标注的 10K-100K 高质量 Q&A 对 | 在 base model 上继续训练（有监督） | 仅类比：像教一个知识渊博但不会说话的人学对话礼仪 |
| **RLHF** | 人类对多个回答的偏好排序 → reward model → PPO | 在 SFT model 上用 RL 优化 | 仅类比：像大众评审给选手打分，选手根据分数调整表演 |
| **Function Calling** | SFT 阶段加入工具调用格式的训练样本 | 在 system prompt 里描述可用工具 → 模型生成调用 JSON | 不是"学会用工具"，是"学会了一种特殊的输出格式" |

## 1. 完整训练管线

```
Stage 1: Pretraining           Stage 2: SFT              Stage 3: RLHF
┌─────────────────┐    ┌──────────────────┐    ┌──────────────────────┐
│ 10TB 互联网文本   │ →  │ ~100K 对话样本    │ →  │ 人类偏好排序          │
│ 目标: 预测 token  │    │ 目标: 学对话格式   │    │ 目标: 输出更符合偏好   │
│ GPU: 数千×数周    │    │ GPU: 几十×几天    │    │ 方法: Reward Model + PPO│
│ 产出: Base Model │    │ 产出: SFT Model  │    │ 产出: RLHF Model     │
└─────────────────┘    └──────────────────┘    └──────────────────────┘
  ↓ 只会补全文本          ↓ 会对话了               ↓ 回答更有帮助、更安全
```

## 2. Stage 2: SFT 深入

### 为什么 SFT 只需要少量数据？

```
预训练: 10TB 文本 — 学会了"语言是什么"（知识 + 语法 + 推理模式）
SFT:    100K 样本 — 只需要学会"用对话格式展现已有知识"

类比（仅类比）: 
  预训练 = 读了一辈子书
  SFT = 上了一个"如何做PPT展示"的培训班
  书读完了知识就在那儿，培训班教的只是"怎么把知识讲出来"
```

### SFT 怎么教会工具调用（Function Calling）

```json
// SFT 训练样本示例
{
  "messages": [
    {"role": "user", "content": "明天北京的天气怎么样？"},
    {"role": "assistant", "content": null, 
     "function_call": {"name": "get_weather", "arguments": "{\"city\": \"北京\"}"}}
  ]
}
```

> [!IMPORTANT]
> **工具调用不是"理解"了工具，而是学会了一种输出格式。** 模型在 SFT 阶段看到大量"当用户问 X 时，输出 `function_call` JSON"的样本，学会了在合适的时候生成这种格式。Muse 的 MCP 工具（notify_planner、search_memory）就是靠这个机制工作的。

## 3. Stage 3: RLHF 深入

### Reward Model 怎么训练的

```
1. 给 SFT Model 一个问题
2. 模型生成 4 个不同的回答 (A, B, C, D)
3. 人类标注员排序: B > D > A > C
4. 用这些排序数据训练 Reward Model
   - 输入: (问题, 回答) 
   - 输出: 一个分数 (越高越好)
5. Reward Model 学会了"什么样的回答人类更喜欢"
```

### PPO 怎么优化的

```
循环:
  1. 模型生成一个回答
  2. Reward Model 给这个回答打分
  3. 如果分高 → 增加生成这种回答的概率
     如果分低 → 降低生成这种回答的概率
  4. 同时加一个 KL 散度约束 → 不能偏离 SFT Model 太远（防止忘记能力）
```

> [!IMPORTANT]
> **RLHF 的核心权衡：** 太少则模型不够有帮助，太多则模型变得"过度服从"（sycophantic）— 只会说你想听的话，而不是真的有用的话。这是当前 LLM 对齐研究的核心难题。

## 4. 推理阶段的关键参数

| 参数 | 作用 | Muse 怎么用 |
|------|------|-----------|
| **temperature** | 控制采样随机性。0=确定性，1=创意 | pua 聊天: 0.7-0.9 / Workflow 执行: 0.1 |
| **top_p** | 只从概率最高的前 p% token 中采样 | 通常 0.9-0.95 |
| **max_tokens** | 最大输出长度 | 控制成本和延迟 |
| **stop sequences** | 指定停止生成的标记 | 工具调用结束标记 |

## 💼 面试必答

**Q: SFT 和 RLHF 的区别是什么？**
> SFT 用有监督的方式教模型对话格式（怎么回答），RLHF 用强化学习教模型人类偏好（回答得更好）。SFT 改行为模式，RLHF 优化行为质量。

**Q: Function Calling 是怎么实现的？**
> 在 SFT 阶段，训练样本中包含工具调用格式的 JSON 输出。模型学会了在合适的语境下生成 `function_call` 格式，而不是自然语言。这不是模型"理解"了工具，而是学会了一种特殊的输出格式。

**Q: 为什么 SFT 只需要少量数据？**
> 因为知识已经在预训练阶段编码在参数中了。SFT 只需要教模型"用对话格式展现已有知识"，这个映射关系比原始知识简单得多，所以需要的数据量也少得多。
