## 🔬 核心原理（三栏表）

| 概念 | 能力来源（模型怎么获得的） | 激活方式（工程上怎么触发） | 类比（仅类比） |
|------|------------------------|------------------------|--------------| 
| **Next-token prediction** | 在海量文本上训练：给定前 N 个 token，预测第 N+1 个 | 自回归生成：每次预测一个 token，追加到序列末尾，再预测下一个 | 仅类比：像"接龙游戏"，但每次只接一个字 |
| **Transformer** | 架构创新（Vaswani 2017）：Self-Attention 让模型看到所有位置的信息 | 模型前向传播时，每层 attention 计算 token 之间的关联权重 | 仅类比：像一个教室里每个学生都能直接和其他任何学生交流 |
| **参数 = 压缩的知识** | 预训练阶段：10TB 文本压缩成 ~140GB 参数文件（LLaMA 70B） | 参数是固定的权重矩阵，推理时不变 | 仅类比：像把整个维基百科"蒸馏"成一个文件 |
| **CoT 推理** | **Prompt CoT** [ref-08]: 预训练语料含推理과정, 用 prompt 激活。**Training CoT** [ref-05]: GRPO 强化学习, reward 奖励"思考后答对", 模型自动学会 `<think>` | Prompt: "Let's think step by step"。Training: 模型自动在回答前产出 `<think>` 推理过程 | 仅类比：像考试写解题步骤（但 Training CoT 是 RL 训练涌现的，不是人教的） |
| **Tool Use** | SFT 阶段用工具调用样本训练 | Function Calling 格式 + 系统 prompt 定义工具 | 仅类比：像人拿起计算器辅助计算 |

## 1. LLM 到底是什么？

> **来源: [C1] Karpathy Intro to LLMs + [B1] Raschka ch01**
> **repos/ 参考: `repos/LLMs-from-scratch/ch01/` — 详细解释 LLM 定义、发展、应用**

### 1.1 两个文件

Karpathy 的开场非常直接：

> 一个 LLM 就是两个文件：
> 1. **参数文件**（parameters）— 一堆浮点数，比如 LLaMA 70B 就是 140GB
> 2. **运行代码**（run file）— 大约 500 行 C 代码就能跑起来

**这意味着什么？**
- LLM 可以完全离线运行（不需要网络）
- 参数文件 = 从训练数据中"压缩"出来的知识
- 运行代码做的事就是：给定已有的 token 序列，预测下一个 token

### 1.2 Next-Token Prediction 的深层含义

> [!IMPORTANT]
> **预测下一个 token ≠ 简单的自动补全。**
> 如果模型能准确预测任何文本的下一个 token，它必须"理解"文本的含义。
> 比如预测一道数学题答案的下一个数字，模型必须"会做"这道题。

```
输入: "法国的首都是"
       ↓ 模型计算所有可能 token 的概率
预测: "巴" (概率 0.95) > "柏" (概率 0.01) > ...
       ↓ 采样（可能是 top-p / temperature 控制）
输出: "巴"
       ↓ 追加到序列
输入: "法国的首都是巴"
       ↓
预测: "黎" (概率 0.99)
...
```

**为什么这这么强大？**

要准确预测下一个 token，模型被迫在参数中编码：
- 语法规则（"is" 后面该跟什么词性）
- 事实知识（法国首都是巴黎）
- 推理能力（数学题的答案是什么）
- 编程能力（这段代码下一行该写什么）

> Karpathy: "Next-token prediction is a deceptively simple objective, but it turns out to be extremely powerful."

## 1.5 大模型发展关键拐点

> **为什么要讲这个？** 你必须知道这条发展线——面试一定会问，而且理解"每个拐点解决了什么问题"能帮你判断技术趋势。

### 发展时间线

```
2017  Transformer 论文 (Vaswani et al.)
  ↓   解决了什么: RNN 无法并行训练 → Attention 机制实现全局并行
  
2018  GPT-1 (1.17 亿参数)
  ↓   突破: 证明了"预训练 + 微调"范式可行
  
2019  GPT-2 (15 亿参数)
  ↓   突破: 规模放大 → 零样本学习涌现（不用微调也能做任务）
  
2020  GPT-3 (1750 亿参数)
  ↓   突破: In-context Learning (上下文学习) — 给几个例子就能学会新任务
  ↓   Scaling Law: 参数越多 + 数据越多 = 能力越强（可预测）

2022  ChatGPT (GPT-3.5 + RLHF)
  ↓   突破: RLHF 让模型从"会写文本"变成"会对话"
  ↓   这是 SFT + RLHF 的工程胜利，不是模型架构的胜利

2023  GPT-4 (未公开参数)
  ↓   突破: 多模态（文字+图片）+ 更强推理
  ↓   Claude-2, LLaMA-2 等开源追赶

2024  DeepSeek R1 + OpenAI o1
  ↓   突破: 思考链 (CoT) 从 prompt 技巧 → RL 训练出来的内生能力
  ↓   Test-time Compute Scaling: 推理时多花计算 = 更准确

2025  Claude 4 + GPT-4.5 + Gemini 2
  ↓   趋势: Agent 能力内建（Tool Use / Computer Use 原生支持）
  ↓   趋势: 长上下文 (128K-1M tokens)
```

### 3 个关键拐点（面试重点）

| 拐点 | 之前 | 之后 | 业界影响 |
|------|------|------|---------|
| **Scaling Law (2020)** | 不知道模型该多大 | 参数 × 数据 × 计算 → 性能可预测 | 催生了"军备竞赛"——GPT-4 训练花 $100M+ |
| **RLHF (2022)** | Base Model 只会续写文本 | 会对话、听指令、拒绝有害请求 | ChatGPT 爆火，AI 进入大众视野 |
| **Test-time Scaling (2024)** | 推理速度固定 | 难题花更多 token 思考 = 更高准确率 | DeepSeek R1 / o1 出现"思考过程" |

> [!IMPORTANT]
> **面试怎么说：** "大模型发展有 3 个关键拐点。第一个是 Scaling Law，证明了模型越大越好且可预测。第二个是 RLHF，把文本补全器变成了助手。第三个是 test-time compute scaling，让模型在难题上花更多计算来提高准确率——这就是 DeepSeek R1 的 `<think>` 标签和 OpenAI o1 的 '推理模式' 背后的原理。"

## 2. 训练三阶段

> **来源: [C3] Karpathy State of GPT + [B1] Raschka ch05-07 + [P3] InstructGPT 论文**
> **repos/ 参考: `repos/LLMs-from-scratch/ch05/` (预训练) + `ch07/` (SFT+DPO)**

### 2.1 Stage 1: 预训练（Pretraining）

| 维度 | 细节 |
|------|------|
| **数据** | ~10TB 文本（互联网爬虫 + 书籍 + 代码 + 维基百科） |
| **计算** | 数千 GPU × 数周 = 数百万美元 |
| **目标** | Next-token prediction（自监督） |
| **产出** | **Base Model** — 能写文本，但不会对话，不听指令 |

**Base Model 的特征：**
- 给它 "法国的首都是"，它会接 "巴黎"
- 给它 "请帮我写一封邮件"，它可能会接 "请帮我写一份报告" 而不是真的写
- 它不知道自己是一个"助手"，它只是一个文本补全器

### 2.2 Stage 2: 监督微调（SFT — Supervised Fine-Tuning）

| 维度 | 细节 |
|------|------|
| **数据** | ~10K-100K 高质量对话样本（人类写的 Q&A） |
| **目标** | 让 Base Model 学会"问答"格式 |
| **产出** | **SFT Model** — 会对话了，但可能说有害内容 |

**SFT 做了什么？**
```
训练样本:
  User: "什么是光合作用？"
  Assistant: "光合作用是植物利用光能将二氧化碳和水转化为葡萄糖和氧气的过程。"

模型学会: 看到 "User:" 后面的问题 → 生成 "Assistant:" 后面的回答
```

> [!IMPORTANT]
> **SFT 改变的是模型的"行为模式"，不是"知识量"。** 知识在预训练时已经有了，SFT 只是教模型怎么用对话格式来展现知识。

**工具调用也是 SFT 教的：**
```
训练样本:
  User: "今天北京天气怎么样？"
  Assistant: <tool_call>{"name": "get_weather", "args": {"city": "北京"}}</tool_call>

模型学会: 需要实时信息时 → 输出特定格式的工具调用
```
→ 这就是 Claude Code 能调工具的基础：它在 SFT 阶段见过大量工具调用样本。

### 2.3 Stage 3: RLHF（Reinforcement Learning from Human Feedback）

| 维度 | 细节 |
|------|------|
| **数据** | 人类对多个回答的偏好排序 |
| **方法** | 训练 reward model → 用 PPO 优化生成策略 |
| **产出** | **RLHF Model** — 更有帮助、更安全、更符合人类偏好 |

**RLHF 的流程：**
```
1. 给模型同一个问题，让它生成多个回答 (A, B, C, D)
2. 人类标注员排序: B > D > A > C
3. 用这些排序训练一个 Reward Model（打分器）
4. 用 Reward Model 指导主模型优化（PPO 算法）
5. 结果：模型生成的回答越来越符合人类偏好
```

## 3. LLM 的能力边界（面试重点）

### 3.1 LLM 能做什么

| 能力 | 来源 | 注意 |
|------|------|------|
| 语言理解和生成 | 预训练 | 核心能力 |
| 知识问答 | 预训练（编码在参数中） | 有截止日期，可能过时 |
| 代码生成 | 预训练（训练数据含代码） | 能力强但不保证正确 |
| 推理（CoT） | 预训练 + prompt 激活 + RL 强化 | 后面 §4 详细讲 |
| 遵循指令 | SFT 阶段训练 | 核心行为 |
| 工具使用 | SFT 阶段的工具调用样本 | Function Calling |
| 安全对齐 | RLHF 阶段 | 可被越狱 |

### 3.2 LLM 不能做什么

| 限制 | 原因 | 怎么绕过 |
|------|------|---------|
| **不会真的"想"** | 每个 token 的计算量是固定的（1 次前向传播） | CoT: 通过多 token 输出来分配更多计算 |
| **会幻觉** | 训练目标是"看起来像真的"，不是"是真的" | RAG: 用检索结果约束 |
| **有知识截止日** | 参数在训练后固定 | Tool Use: 调搜索引擎 |
| **上下文有限** | Transformer 的 O(n²) 注意力 | 压缩/摘要/外部存储 |
| **不是确定性的** | 采样有随机性（temperature） | temperature=0 近似确定性 |

> [!IMPORTANT]
> **面试回答模板：** "LLM 本质是一个通过 next-token prediction 训练的神经网络。它不是在'思考'，而是在基于训练数据的统计规律预测最可能的下一个 token。它的'推理能力'是通过 prompt 引导（如 CoT）让模型在输出中花更多 token 来分解问题，从而提高准确率。"

## 4. 思考和规划是怎么实现的？

> **这是面试高频题。** 别人问你"给 Claude Code 发消息为什么会出现思考？"你得能讲清楚。

### 4.1 思考能力的 3 层进化

```
层级 1: Prompt CoT（2022）       ← 纯工程技巧，模型没变
层级 2: 训练出的 CoT（2024）     ← RL 训练，模型内生变化
层级 3: Agent Loop（2025）       ← 系统工程，循环调用模型

↑ 每一层建立在前一层之上
```

### 4.2 层级 1: Prompt CoT — 纯工程技巧

**时间：** 2022 (Wei et al.)

**原理：** 模型本身没有"思考能力"。但预训练语料中包含大量解题过程（数学课本、编程教程），模型学会了"解题步骤"这种文本模式。通过 prompt 提示，可以激活这种模式。

**具体例子：**
```
❌ 直接问（不引导思考）:
  User: "23 × 47 = ?"
  Assistant: "1071"  ← 1次前向传播直接猜答案，大概率错

✅ 引导思考（Prompt CoT）:
  User: "23 × 47 = ? 请一步一步计算。"
  Assistant: 
    "Step 1: 23 × 40 = 920     ← 简单乘法，1次前向传播够了
     Step 2: 23 × 7 = 161      ← 简单乘法，1次前向传播够了
     Step 3: 920 + 161 = 1081   ← 简单加法，1次前向传播够了
     答案: 1081"                 ← 最终答案正确
```

**为什么有效（第一性原理）：**
- LLM 每预测一个 token 用的计算量是**固定的**（一次前向传播）
- 复杂问题需要的计算量 > 一次前向传播
- CoT 把一个复杂问题拆成 N 个简单子问题
- 每个子问题一次前向传播就够 → 每步对的概率高
- N 步都对 → 最终答案对的概率大大提高
- **本质：用更多 token（输出更多文字）= 分配更多计算**

### 4.3 层级 2: 训练出的 CoT — DeepSeek R1 & OpenAI o1

> **来源: [ref-05] DeepSeek R1 论文** (arxiv 2501.12948)
> **来源: [ref-09] philschmid Mini-R1 教程** (philschmid.de/mini-r1)
> **来源: [ref-14] huggingface/open-r1** (GitHub 复现项目)

**时间：** 2024

**关键区别：** 层级 1 是你告诉模型"请一步步想"，层级 2 是模型**自己学会了先想再答**。

#### GRPO 到底是什么？用一个具体例子讲

> **来源: [ref-05] DeepSeek R1 论文 §2.2**

传统 RLHF 用 PPO 算法，需要一个额外的"评分模型"（Critic），和主模型一样大 → 训练成本翻倍。DeepSeek 发明了 **GRPO (Group Relative Policy Optimization)**，核心创新是：**不需要评分模型，让一组回答互相比较。**

```
具体例子 — 训练 R1 解"鸡兔同笼"：

第 1 步: 出题
  问题: "笼子里有鸡和兔共 35 只，脚共 94 只，问鸡兔各多少？"

第 2 步: 模型生成一组回答 (比如 G=8 个)
  回答 A: <think>设鸡x兔y，x+y=35，2x+4y=94，解得x=23,y=12</think> 鸡23兔12 ✅
  回答 B: <think>鸡有2脚兔有4脚...嗯...大概鸡20兔15</think> 鸡20兔15    ❌
  回答 C: <think>总共94脚，如果全是鸡70脚，差24脚，24÷2=12只兔</think> 鸡23兔12 ✅
  回答 D: 鸡25兔10 ← 没思考，直接猜                                    ❌
  回答 E: <think>设兔=y, 2(35-y)+4y=94, 70+2y=94, y=12</think> 鸡23兔12  ✅
  ... (共 8 个)

第 3 步: 规则打分 (不需要人，也不需要评分模型!)
  答案对了 = reward 1, 错了 = reward 0
  A=1, B=0, C=1, D=0, E=1, F=0, G=1, H=0

第 4 步: 计算"相对优势" (GRPO 的核心)
  组平均 reward = 0.5
  A 的优势 = (1 - 0.5) / σ = 正的 → 增强这种策略
  B 的优势 = (0 - 0.5) / σ = 负的 → 削弱这种策略
  D 的优势 = 负的 → "直接猜"这种行为被惩罚

第 5 步: 更新模型
  模型学到: "先在 <think> 里推理 → 答案对的概率更高 → 以后多推理"
  模型也学到: "直接猜 → reward 低 → 以后少猜"
```

**GRPO vs PPO 的本质区别：**

| 对比 | PPO (传统) | GRPO (DeepSeek) |
|------|-----------|-----------------|
| 评分方式 | 需要一个独立的 Critic 模型打分 | 组内互相比较，谁比平均分高谁就好 |
| 成本 | 主模型 + Critic 模型 = 2 倍显存 | 只有主模型，省一半显存 |
| 打分来源 | Critic 模型的预测（可能不准） | 规则判定（数学题对错、代码能否运行） |
| 稳定性 | Critic 不准时容易跑偏 | 规则打分很稳定 |

> **来源: [ref-09] philschmid Mini-R1** 用一个"Countdown Game"（凑数游戏）在小模型上复现了 GRPO 训练，验证了上述流程是可行的。代码在 `reference/repos/open-r1/` 可以跑。

#### "Aha Moment" — 模型自己学会了自我纠错

> **来源: [ref-05] DeepSeek R1 论文 §2.3.1, Figure 3**
> 这是论文最震撼的发现。

训练到中期，R1-Zero（纯 RL 训练，没有任何 SFT）的输出中自发出现了这样的行为：

```
模型输出 (真实示例，摘自论文 Figure 3):

<think>
Wait, wait. Wait. 
I made an error in the previous step. 
Let me reconsider...

Hmm actually, the approach I was using can't work because... 
Let me try a different method.

(重新推理)

Ah, I see! The answer should be...
</think>

最终答案: ...
```

> [!IMPORTANT]
> **这不是人教的！** 没有任何训练样本包含"等等，我前面错了"这种自我纠错。这是纯 RL 过程中**涌现**的：模型发现"纠错后再答 → reward 更高"，就自动学会了这个行为。论文把这叫 "Aha Moment"。

#### 完整训练流水线（4 阶段）

> **来源: [ref-05] DeepSeek R1 论文 §2.4**

```
阶段 1: 冷启动 SFT
  ├─ 用少量高质量 <think>...</think> 样本做 SFT
  ├─ 目的: 教会模型 <think> 标签的格式 + 基本推理习惯
  └─ 解决问题: 纯 RL 从零开始太不稳定（输出乱码、语言混杂）

阶段 2: 推理 RL (GRPO)
  ├─ 只在数学/代码等有明确对错的任务上训练
  ├─ 规则打分: 答案对=1, 错=0 (不需要人标注)
  ├─ 训练 ~数千步后涌现 Aha Moment
  └─ 产出: 推理能力很强，但其他能力（聊天/写作）可能退化

阶段 3: 拒绝采样 + SFT  
  ├─ 用阶段2的模型在各种任务上生成回答
  ├─ 只保留正确的好回答（拒绝采样）
  ├─ 用这些好回答再做一轮 SFT
  └─ 目的: 恢复写作/聊天等通用能力

阶段 4: 全场景 RL
  ├─ 在有用性/无害性/推理准确性上做最终 RL 优化
  └─ 产出: 最终的 DeepSeek R1 模型

额外: 蒸馏
  ├─ 用 R1 生成的 <think> 过程作为训练数据
  ├─ 训练小模型 (1.5B/7B/27B) 也具备思考能力
  └─ → 这就是 qwen3.5-27b 蒸馏版能"思考"的原理
  └─ 来源: [ref-10] HuggingFace Open-R1 复现了蒸馏流程
```

#### OpenAI o1 的"推理模式"

和 DeepSeek R1 思路类似，但实现细节不同：
- o1 的思考过程是**隐藏的**（用户看不到 `<think>` 标签）
- 使用 "test-time compute scaling" — 系统自动决定在这个问题上花多少计算
- 难题 → 生成更长的隐式思考链 → 更准确
- 简单题 → 短思考 → 快速回答

> [!IMPORTANT]
> **面试关键区分：**
> - **Prompt CoT** (2022): 工程技巧。模型没变，你用 prompt 引导它"展示推理过程"
> - **Training CoT** (2024): 训练创新。通过 RL（GRPO/PPO）让模型**自动学会先思考再回答**
> - **本质相同**: 都是"用更多 token 分配更多计算"
> - **关键不同**: Training CoT 的思考质量更高，因为 RL 奖励了"好的思考路径"

### 4.4 层级 3: Agent Loop — Claude Code 为什么编码能力强？

**关键理解：** Claude Code 的强大不仅仅因为模型好，更因为它是一个**Agent 系统**。

```
你发一条消息给 Claude Code: "帮我重构 auth 模块"

这不是一次 LLM 调用。它是一个循环:

Loop 1: 理解任务
  Claude 调用 bash('find . -name "auth*"')  → 找到目标文件
  Claude 调用 file('read src/auth/login.ts') → 读取代码
  Claude 思考: <think>这个模块有 3 个问题: 1... 2... 3...</think>

Loop 2: 制定计划
  Claude 输出: "我打算分 3 步重构: 
    1. 抽取公共接口
    2. 拆分 login/register 
    3. 加测试"
  等你确认 ← Human-in-the-Loop

Loop 3-5: 逐步执行
  Claude 调用 file('edit src/auth/login.ts', ...)  → 改代码
  Claude 调用 bash('npm test')                     → 跑测试
  测试失败 → Claude 读错误日志 → 修复 → 再跑测试 → 通过

Loop 6: 验证完成
  Claude 调用 bash('git diff')  → 检查所有改动
  Claude: "重构完成，所有测试通过。"
```

**它为什么编码能力强？4 个原因：**

| 原因 | 具体机制 | 对比普通聊天 |
|------|---------|------------|
| **1. Agent Loop** | 多次调用模型，每次专注一个子任务 | 普通聊天一次性回答 |
| **2. Tool Use** | 能读文件、改文件、跑命令、看结果 | 普通聊天只能输出文字 |
| **3. 自验证** | 改完代码 → 跑测试 → 失败就修 → 循环 | 普通聊天不知道代码对不对 |
| **4. 上下文管理** | 只读需要的文件，不一次性塞全部代码 | 普通聊天上下文很快用完 |

> [!IMPORTANT]
> **面试怎么讲 Claude Code：**
> "Claude Code 的编码能力强，不是因为模型 '特别聪明'，而是因为它是一个 Agent 系统。它把一个复杂任务拆成多个简单子任务（循环），每个子任务模型都能做好。加上 Tool Use（读写文件、跑命令）和自验证（跑测试检查结果），它形成了一个 Reason → Act → Observe 的闭环。这就是 unit01 学的 Agent 核心循环的最佳实践。"

## 💼 面试必答

**Q: LLM 是怎么工作的？**
> 本质是 next-token prediction。在海量文本上预训练，学会预测任意文本序列的下一个 token。然后通过 SFT（监督微调）学会对话格式和工具调用，通过 RLHF 学会符合人类偏好。

**Q: 大模型发展有哪些关键拐点？**
> 三个拐点：2020 年 Scaling Law 证明了"大就是好"且可预测；2022 年 RLHF（ChatGPT）把文本补全器变成助手；2024 年 test-time compute scaling（DeepSeek R1/o1）让模型学会"先想再答"。

**Q: DeepSeek R1 的"思考"是怎么实现的？**
> 4 步：先用少量样本 SFT 教会 `<think>` 格式；然后用 GRPO（一种高效 RL 算法）让模型生成多个回答，正确答案获得高 reward；模型发现"先思考 → 答案更对"；最终自发学会了自我纠错。训练后期出现了 "Aha Moment"——模型自己说"等等，我前面错了"。最后通过蒸馏把思考能力传到小模型。

**Q: CoT 为什么有效？**
> 本质是计算量分配。LLM 每生成一个 token 的计算量是固定的（一次前向传播），复杂问题需要的计算量超过一次前向传播。CoT 把问题拆成 N 个简单子问题，每个子问题一次前向传播就够了。"更多 token = 更多计算 = 更高准确率"。

**Q: Claude Code 为什么编码能力强？**
> 不是因为模型特别聪明，而是 Agent 系统设计好。它用 Agent Loop（Reason → Act → Observe 循环），能调工具读写文件和跑命令，改完代码会自动跑测试验证，失败了就修到通过。这把一个复杂的编码任务拆成了多个模型能轻松完成的小任务。

**Q: Prompt CoT 和 Training CoT 有什么区别？**
> Prompt CoT 是工程技巧——你在 prompt 里说"一步步想"，模型听了就展示推理过程，模型本身没变。Training CoT 是训练创新——通过 RL（如 GRPO）让模型在训练中自己学会了"先思考再回答"，思考质量更高。本质都是"更多 token = 更多计算"，但 Training CoT 的思考更有方向性，因为 RL 奖励了好的推理路径。

1. **LLM 的 "理解" 是真的理解吗？** 从 next-token prediction 角度解释。
2. **SFT 和 RLHF 分别改变了模型的什么？** 用一句话分别说清楚。
3. **给你一个场景：用户问 "15 × 17"，模型直接回答 vs CoT 回答，准确率差多少？为什么？**
4. **DeepSeek R1 的 `<think>` 标签是 prompt 引导的还是训练出来的？** 具体训练流程是什么？
5. **如果面试官问 "为什么 Claude Code 比普通 ChatGPT 编码能力强？"，你怎么回答？**

<details>
<summary>参考答案</summary>

1. 不是人类意义上的"理解"。模型在参数中编码了文本的统计规律。预测"法国的首都是__"后面是"巴黎"，不是因为模型"知道"巴黎是法国首都，而是因为训练数据中这个模式出现频率极高。但这种统计能力足够强大时，行为上很难和"理解"区分。

2. SFT 改变**行为模式**（从续写文本 → 遵循指令对话），不改知识。RLHF 改变**输出偏好**（从"看起来对"→"符合人类价值观"），是价值观对齐。

3. 直接回答准确率很低（可能 30%-50%），CoT 可以到 90%+。因为 15×17 的计算量超过一次前向传播，直接猜大概率错。CoT 拆成 15×10=150、15×7=105、150+105=255，每步的计算量一次前向传播就够了。

4. 训练出来的。流程：冷启动 SFT 教格式 → GRPO 强化学习（生成 N 个回答，正确的得高 reward，模型发现"先想→答案更对"）→ 出现自我纠错的 Aha Moment → 蒸馏到小模型。

5. "4 个原因：Agent Loop 把大任务拆小、Tool Use 能读写文件和跑命令、自验证（改完跑测试）、上下文管理（只读需要的文件）。本质是 Agent 系统设计，不是模型本身特别强。"

</details>
