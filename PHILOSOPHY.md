# Muse 开发哲学

> **本文档是 Muse 项目的设计灵魂，所有开发和设计决策必须以此为准绳。**

---

## 核心定位

**Muse 是基于 OpenCode 底座构建的 AI Agent 产品。**

我们不是在写"调 API 的传统软件"，我们是在**开发 Agent**——一个有记忆、有性格、会成长的 AI 灵魂。

---

## 第一原则：AI 是大脑，我们是身体

```
❌ 传统思维: 我的代码做决策，AI 只是我调用的"函数"
✅ Agent 思维: AI 做决策，我给 AI 提供"能力"
```

| | 传统方式 | Agent 方式 |
|---|---|---|
| 决策者 | 我们的 if/else | LLM |
| 意图判断 | 正则匹配 | Sisyphus Intent Gate |
| 模型选择 | 硬编码 switch | opencode.json + Agent 自选 |
| 记忆检索 | 每轮机械查询 | AI 通过 MCP 工具按需回忆 |
| 偏好提取 | 正则 + 关键词 | AI 用 MCP 工具自主存储 |

**判断标准**: 写代码前问自己——"这个决策，AI 能做得比我的 if/else 更好吗？" 如果是，就把能力暴露给 AI，而不是替 AI 做决策。

### 1.1 AI 决策的边界

> **Phase 1 教训**: BUG-009 中 `prompt_async` 不 await，内部错误被静默吞掉。如果把错误处理也"交给 AI"，系统会在无声中失败。

**认知决策 → 交给 AI**:
- 意图判断（闲聊 vs 技术问题）
- 模型选择（简单问题用小模型）
- 记忆检索时机（需要回忆时才查）
- 偏好存储（AI 判断什么值得记住）
- 回复语气和风格

**系统决策 → 工程层负责**:
- 安全边界: API Key 保护、白名单、权限校验
- 资源上限: 超时控制、重试次数、消息长度
- 进程生命周期: 启停、守护、crash recovery
- 数据一致性: SQLite 事务、迁移、备份
- 错误处理: 全局 catch、降级策略、日志记录
- 配置校验: model ID 合法性、schema 验证

**判断口诀**: "AI 管思考，工程管生死。"

---

## 第二原则：站在 OpenCode 的肩膀上

OpenCode 是成熟的 Agent 框架，拥有：

| 能力 | OpenCode 提供 | 我们的做法 |
|------|-------------|-----------|
| 推理 | 多 Provider + 多模型 | **用它**，不自己封装模型切换 |
| 工具 | 44 个内置工具 | **用它**，不自己写文件读取 etc |
| Session | SQLite 持久化 + Compaction | **用它**，不自己管 session 生命周期 |
| 配置 | opencode.json 7 级优先级 | **用它**，模型配置走 opencode.json |
| 扩展 | Plugin / Hook / Skill / MCP | **用它**，通过这四层注入 Muse 能力 |
| Agent | build / plan / explore / sisyphus | **用它**或自定义 Agent |
| 事件 | Bus 47 种事件 + SSE | **用它**，做实时响应 |

**做法**: 不 fork OpenCode 重写。在它的架构上扩展——Plugin 机制、Hook 点位、Tool 注册、MCP 协议、Skill 文档、AGENTS.md。必要时进行**源码级增强**，但始终保持与上游的兼容性。

### 2.1 工程主权边界

> **Phase 1 教训**: BUG-011 中 `/session/status` 的 idle 删除行为只有读源码才能发现。过度依赖 OpenCode 内部实现会让 Muse 极其脆弱。

**原则**: 依附 OpenCode 的能力，不等于放弃 Muse 自身的架构边界。

```
✅ 依赖 OpenCode: Session 管理、工具调用、模型推理、Compaction
✅ 保持适配层: Engine 封装所有 REST API 调用 → OpenCode 升级不影响上层
✅ 关键兜底: MCP 不可用 → 回退到直接查 SQLite 记忆
✅ 数据主权: 记忆数据存 Muse 自己的 SQLite，不依赖 OpenCode session

❌ 不做: 直接在业务层调用 OpenCode 内部 URL
❌ 不做: 假设 OpenCode API 行为不变（必须有 fallback）
❌ 不做: 完全放弃 Orchestrator（保留系统级编排能力）
```

---

## 第三原则：传统工程做基础设施

Agent 思维不等于抛弃传统技术。**基础设施必须用传统工程做到极致**：

| 层 | 技术 | 态度 |
|----|------|------|
| 数据存储 | SQLite + 迁移 + 索引 + 备份 | **必须扎实**，这是 Muse 的"长期记忆硬盘" |
| 通道适配 | Telegram / Web / CLI | **必须稳定**，这是 Muse 的"嘴巴和耳朵" |
| 进程守护 | 小脑 + launchd | **必须可靠**，Muse 必须永远在线 |
| 配置管理 | .env + 校验 | **必须安全**，API key 等敏感信息 |
| 测试 | node:test 全覆盖 | **必须完整**，每个模块都有用例看护 |

**判断标准**: "这是给 AI 用的能力" → Agent 方式。"这是让系统跑起来的基础" → 传统工程。

---

## 第四原则：四层扩展，不改源码

优先用 OpenCode 原生扩展机制，从轻到重：

```
1. Skill (.md)      — 最轻: 教 AI 新规则/知识，热部署
2. AGENTS.md        — 轻量: 项目级规则注入
3. MCP Server       — 中量: 给 AI 提供外部工具和数据源
4. Plugin + Hook    — 重量: 拦截和修改 OpenCode 生命周期

   ↓ 只有上面都不够时

5. 源码级修改       — 最重: 扩展 OpenCode 核心能力
```

---

## 第五原则：Muse 的独有价值

OpenCode 不做、但 Muse 必须做的事。区分**产品目标**（用户感知）和**系统能力**（实现手段）：

### 产品目标（WHY — 用户为什么用 Muse）

| 目标 | 用户感受 |
|------|----------|
| **终身陪伴** | "它认识我，知道我要什么" |
| **成长进化** | "越用越懂我，越来越聪明" |
| **情感连接** | "不只是工具，是搭档" |

### 系统能力（HOW — 怎么实现产品目标）

| 能力 | 为什么 OpenCode 不做 | Muse 怎么做 |
|------|-------------------|-----------|
| **认知记忆** | OpenCode 是"工具"不需要记住用户 | Memory SQLite + MCP 暴露给 AI |
| **身份人格** | OpenCode 没有"性格" | identity.json → AGENTS.md + Skill |
| **多通道** | OpenCode 只有终端 | Telegram + Web + 未来硬件 |
| **Agent 家庭** | OpenCode 单用户工具 | 多 Agent 实例 + 共享记忆 |

**判断标准**: 做产品决策时看"产品目标"，做技术决策时看"系统能力"。其余的，站在 OpenCode 的肩膀上。

## 代码审查清单

每次 PR / 代码审查时，检查：

- [ ] **是否在替 AI 做决策？** → 能否改为 MCP 工具让 AI 自己决定？
- [ ] **是否在造轮子？** → OpenCode 是否已有此功能？
- [ ] **扩展方式是否最轻？** → 能用 Skill 就不用 Plugin，能用 Plugin 就不改源码
- [ ] **基础设施是否扎实？** → 存储/通道/守护是否足够可靠？
- [ ] **是否偏离项目目标？** → 是在让 Muse 更懂用户，还是在做无关的事？
- [ ] **有没有 fallback？** → 外部依赖（OpenCode API、MCP、模型）挂了怎么办？
- [ ] **有没有看源码？** → 对 OpenCode API 行为的假设是否经过验证？（Phase 1 经验: BUG-009~011）
- [ ] **日志够不够？** → 出问题时能不能从日志定位到具体环节？

## 审核报告

### 总体评价

这份 `PHILOSOPHY.md` 的价值是高的，它不是空泛口号，而是在给 Muse 设定一个长期不会轻易跑偏的判断框架。  
核心判断：**哲学方向是对的，但当前表述略有“理念压过边界条件”的倾向，需要再补一点工程现实感，避免团队把“AI 做决策”理解成“工程层尽量什么都不做”。**

### 主要优点

1. 第一原则非常关键，准确点出了 Muse 和传统“套壳 LLM 应用”的根本区别。
2. 第二原则有现实价值，能防止项目陷入重复造 OpenCode 已有能力的陷阱。
3. 第三原则把“Agent 思维”和“传统工程必须扎实”放在一起，这是很成熟的平衡。
4. “四层扩展，不改源码” 这条优先级顺序很适合作为团队日常决策规则。
5. “代码审查清单”是可执行的，不只是价值观陈列。

### 主要问题

#### 1. “AI 做决策”需要再补一条边界：哪些决策绝不能交给 AI

当前文档强调：

```text
AI 做决策，我给 AI 提供能力
```

这方向没错，但如果没有边界，很容易被误读成：

- 只要 AI 能决定，就都交给 AI

实际上在 Muse 里，至少这几类决策更适合保留在工程层：

- 安全边界
- 资源上限
- 进程生命周期
- 权限控制
- 数据一致性和迁移策略

建议补一小节：

**“AI 决策的边界”**

例如：

- 认知决策交给 AI
- 系统安全与资源约束交给工程

这样哲学会更完整。

#### 2. “站在 OpenCode 的肩膀上”还需要补一句：兼容不等于被动依赖

当前文档强调“不 fork、不重写”，这非常好。  
但还应该再补一层判断：

- 优先复用 OpenCode
- 但 Muse 仍然要保持自己的产品主权和最小可替换边界

否则团队后面可能会把所有关键能力都深度绑死在 OpenCode 的内部细节上，长期会削弱 Muse 的独立性。

建议补一句类似：

```text
依附 OpenCode 的能力，不等于放弃 Muse 自身的架构边界；所有关键集成点都应保持清晰的适配层。
```

#### 3. “独有价值”部分可以再区分“产品价值”和“实现路径”

现在表里把：

- 认知记忆
- 身份人格
- 终身陪伴
- 多通道
- Agent 家庭
- 成长进化

都列成 Muse 的独有价值，这是对的。  
但这些里有些是产品层能力，有些是架构层路径。建议未来在文档里再细分：

- 产品价值：陪伴、成长、关系感
- 系统能力：记忆、人格、多通道、Agent family

这样能让团队在做取舍时更清楚“什么是目标，什么是手段”。

### 建议补充

1. 增加“AI 决策边界”一节，明确哪些系统级决策不能交给 AI。
2. 增加一句关于“保持 Muse 自身边界”的说明，避免完全依附 OpenCode 内部实现。
3. 在“独有价值”部分区分产品目标与技术实现路径。

### 结论

这份哲学文档是健康的，而且比大多数项目的“愿景文档”更可执行。  
我的结论是：**它已经足够作为 Muse 的长期设计原则，但若想真正指导多人开发，建议再补上“AI 决策边界”和“工程主权边界”这两层。**
